# Markdown 写作

这是悟帆内置的 Markdown 写作指南。核心信念:**一份好的 Markdown,是写给人读的,不是写给解析器看的。**

很多模型在产出 `.md` 文件时会落入两个陷阱:

1. **无脑堆标题和列表**——把任何内容都拆成"## 一、## 二、## 三",用项目符号取代完整的句子,让文档变得破碎、失去节奏。
2. **不分场合滥用图表或 emoji**——在一篇日记里插柱状图,在一份技术文档里塞满装饰性符号,两头都不讨好。

本技能的目标是帮 Agent 做到三件事:

1. **先判断体裁,再下笔**——散文/日记走文学体,周报/报告走信息体,命令不同、排版不同。
2. **把内容写满,再考虑版式**——段落必须成段,论点必须有论据,不用列表回避思考。
3. **该可视化时可视化,不该图表时绝不图表**——信息体可按需使用 charts 语法,文学体一律保持纯版式。

## 一、先判断体裁

拿到任务后,**第一步不是写,而是判断这是哪一类 Markdown**。不同体裁决定了完全不同的写法。

| 类别 | 典型场景 | 核心诉求 | 是否用 charts |
|------|----------|----------|---------------|
| **文学体** | 日记、散文、随笔、影评、书评、游记、故事、情书、悼文、人物速写、生活记录 | 有呼吸感的文字、情绪的起伏、个人视角 | **不用** |
| **信息体·轻** | 会议纪要、周报、日报、个人笔记、读书笔记、待办清单、碎片想法 | 信息准确、一眼扫清 | 按需少量 |
| **信息体·重** | 研究报告、技术文档、教程、白皮书、产品方案、数据分析、调研总结、竞品对比 | 结构清晰、证据充分、可视化到位 | 按需充分使用 |
| **说明文/规范类** | README、API 文档、使用手册、接口说明、配置指南、FAQ | 结构稳定、可检索、示例完整 | 按需(多用代码块与表格) |
| **商务文案** | 提案、方案、介绍稿、自我简介、项目 pitch | 观点明确、逻辑自洽、可直接用 | 按需 |

判断完之后,**在心里给这篇 Markdown 打一个标签**,例如"这是一篇文学体的散文"或"这是一份信息体·重的技术教程",然后严格按对应的章节要求去写。不要混用体裁。

## 二、通用原则(所有体裁都适用)

### 1. 段落必须是段落

错误示范:

```markdown
今天天气不错。
- 出门散步。
- 看到一只猫。
- 心情很好。
```

这不叫 Markdown,这叫"任务清单伪装成日记"。改成:

```markdown
今天天气不错。午后出门,穿过小区后门的银杏道,看见一只橘黄色的老猫蹲在石凳上,眼睛半闭着,像是在等谁。我停下来,和它对视了一会儿,谁也不开口,心里忽然很安静。
```

**原则**:只要一句话能说清的事,就别分三行列表。只有**真正并列、没有时序或因果关系**的多项信息才用列表。

### 2. 标题用来分章,不用来断气

标题是**骨架**,不是**装饰**。一篇 1000 字的文章塞六个二级标题会让文字支离破碎。

- **短文(<1500 字)**:尽量只用一个一级标题 + 段落,不加二级标题,靠段落自然推进。
- **中等篇幅(1500–4000 字)**:用 2–4 个二级标题组织章节,章节内部靠段落。
- **长文(>4000 字)**:可用二级 + 少量三级标题,但三级标题下至少要有完整段落而不是只有列表。

**绝对禁止**:一级标题下立刻又一个二级标题、中间不写任何过渡文字。

### 3. 排版有呼吸

一屏 Markdown 全是密密麻麻的 `**加粗**` 和 emoji 会让人眼晕。原则:

- **强调要稀缺**:一个段落最多 1–2 处 `**加粗**`,整篇文档 `**加粗**` 不超过 10%。
- **emoji 默认不用**:除非用户明确要求,或确实是情绪表达(如日记尾巴的一个符号),否则纯文字。
- **水平线 `---`**:只在章节级切换(如"正文"到"附录")时使用,不要每段一条。
- **引用块 `>`**:用来引用外部原文、用户原话、名言。不要用来"给段落加边框"。

### 4. 列表要有信息密度

坏例子(空心列表):

```markdown
- 高效
- 稳定
- 易用
```

好例子(每条都有实质内容):

```markdown
- **高效**:单机吞吐 12 万 QPS,相比旧架构提升 3 倍;批处理任务在 8 核环境下平均耗时从 42 分钟下降到 14 分钟。
- **稳定**:灰度运行 90 天,P99 延迟稳定在 120ms 以下,期间零 P0 故障。
- **易用**:对上层业务透明,仅需改 1 行 SDK 调用,现已被 11 个团队接入。
```

**原则**:列表每一条,至少要是一个**完整的、带证据的句子或短段落**。

### 5. 代码块用三个反引号,标语言

```markdown
` ` ` python
def hello():
    print("world")
` ` `
```

(实际写作时不需要空格,此处避免嵌套转义)

标语言是为了让渲染器高亮。常用语言标识:`python` / `bash` / `typescript` / `tsx` / `json` / `yaml` / `markdown` / `sql` / `html` / `css`。

## 三、文学体写作要点(日记、散文、随笔等)

### 禁忌

- **禁止**使用 charts 代码块。散文里插一个柱状图是一种冒犯。
- **禁止**把情感拆成项目符号。情绪需要连贯的语流。
- **禁止**堆叠二级、三级标题。文学体通常一级标题即止,甚至只有正文没有标题。
- **禁止**套用"总—分—总"或"背景/经过/结论"这类报告结构。

### 推荐

- **开头一句就要落地**:不要"本文将探讨……",直接从一个具体的细节、画面或念头切入。
- **用具体代替抽象**:不说"那天很感人",而说"她用围巾擦了擦镜片,镜片上的雾气在灯下散开"。
- **节奏靠长短句**:连续三句长句之后,放一个短句,像深呼吸。
- **结尾留白**:不必强行升华和点题。一句平静的话、一个画面、一个悬而未决的疑问都可以收尾。

### 版式模板

```markdown
# 标题(可省略)

第一段。直接从一个画面、一个动作或一个念头切入,像拉开一扇窗。不要先交代时间地点天气,除非这些本身是意象的一部分。

第二段。展开这个画面或继续这个动作,加入一些感官细节——你闻到了什么,听到了什么,想到了什么。允许有长句。允许有不完整的句子。

第三段。开始有一点转折或反思。但不要用"然而""其实"这类连词把转折做得太明显,让它像水面下的暗涌。

……

最后一段。一句话,或两句话。不点题,只收束。像电影最后一个定格镜头。
```

### 写日记的小技巧

- 可以不写日期标题,或者只在正文第一句带一下("今天是一个没什么事发生的日子。")
- 不必从早写到晚,挑**一个"今天真正发生过什么"的瞬间**展开比流水账强一百倍。
- 允许写未完成的念头、没想通的事、矛盾的情绪。日记的价值在于诚实,不在于完整。

## 四、信息体写作要点(周报、报告、教程、文档)

### 整体结构

不同信息体有不同的主干结构,但都遵循**"先结论,后过程,再细节"**的金字塔。

#### 会议纪要 / 周报

```markdown
# 2026 Q2 第 4 周 周报 · 张三

> 本周核心:完成 A 项目上线、推进 B 项目联调、识别到一个需要关注的风险点。

## 本周进展

- **A 项目上线(已完成)**:灰度 3 天 → 全量,当前日活稳定在 XX,P99 延迟 120ms,暂无 P0 反馈。
- **B 项目联调(进行中,60%)**:已完成登录、权限、核心业务三大接口,剩 2 个边缘接口预计下周三前完成。
- **性能专项(进行中,30%)**:定位到慢查询 TOP10,已优化 3 个,剩余正在和 DBA 评估加索引方案。

## 下周计划

1. 完成 B 项目全部联调并进入预发环境验收。
2. 交付性能专项中期报告。
3. 和产品对齐 C 项目的 Q3 排期。

## 风险 / 需要关注

- **C 项目需求未冻结**:如果本周五前不能冻结,Q3 排期会压缩到不健康的区间。建议:本周三前组织最终评审会议。
```

#### 研究报告 / 调研总结

标准 6 段式:

1. **执行摘要**(TL;DR):3–5 句话讲清楚"我们研究了什么、发现了什么、建议做什么"。
2. **背景与研究方法**:为什么研究、用什么方法研究、数据来源、覆盖范围。
3. **核心发现**:分 3–7 个洞察,每个洞察一段,必要时配 chart。
4. **深度分析**:对关键发现的进一步解释,回答"为什么会这样"。
5. **结论与建议**:把发现转化为可执行的动作。
6. **附录**:参考资料、原始数据、访谈记录等。

#### 教程 / 使用指南

```markdown
# 如何用悟帆搭建一个周报 Agent

- 绑定"会议纪要"和"Markdown 写作"两个技能。
- 配置一条"每周五 18:00 自动生成"的定时任务。

## 前置条件

- 已注册悟帆账号。
- 已在 ……

## 步骤一:创建 Agent

1. 打开 ……,点击右上角 "新建 Agent"。
2. 在 ……

## 步骤二:绑定技能

……

## 常见问题

**Q:定时任务没有触发怎么办?**
A:先检查 ……,再检查 ……

## 下一步

你可以继续尝试 ……
```

### 对"结论先行"的严格执行

信息体最大的毛病是**把结论藏在最后**。用户扫到第三屏还不知道你到底想说什么,体验就崩了。纪律:

- **一级标题之后的第一段**,必须是 2–5 句话的摘要("TL;DR" 或"本周核心"或"核心结论"),直接写结论。
- 不要在前面铺陈"近年来,随着 XX 的快速发展……"这类没信息量的开场白。

### 何时用 charts,何时不用

**用 charts(推荐)**:

- 3 条以上的**数值对比** → `bar` / `table`
- 占比、分布 → `pie`
- 多个维度的能力对比 → `radar` / `comparison`
- 事件演进、阶段划分 → `timeline` / `roadmap`
- 流程、决策、分支 → `flowchart`
- 层级拆解(如竞争力 → 子维度 → 细项) → `mindmap`
- 多个关键数字一起展示 → `metrics`

**不用 charts(强烈建议)**:

- 纯叙事性内容(散文、日记、人物访谈口述)。
- 数据只有 1–2 项(直接用句子/加粗数字讲清楚即可)。
- 用户明确说"简洁""朴素""纯文本""不要图表"。
- 章节内容少于 200 字(图表比例不协调)。

**具体语法**:当你确定要用 chart 时,**先用 `read("skills/markdown-writing/references/charts-syntax.md")` 加载语法参考**(本目录下的 references/charts-syntax.md),再严格按里面的 JSON schema 写。charts 数据**必须来自真实内容**,不可编造。

### emoji 与装饰字符

- **README、博客标题**:可以用 emoji 做视觉锚点(谨慎、每节最多 1 个)。
- **技术文档、研究报告、企业周报**:**不用** emoji,保持专业感。
- **教程**:步骤标题可以用 `1️⃣ 2️⃣` 但不推荐,纯 `## 步骤一` 更稳。
- **日记、散文**:正文一般不用 emoji;如果用,用在句末当成一个情绪符号,一篇不超过 1 个。

## 五、说明文 / 规范类写作

(README、API 文档、使用手册)

### README 最小骨架

```markdown
# 项目名

一句话说明:这个项目是做什么的、解决了什么问题。

## 特性

- 特性一(带一句证据)。
- 特性二。
- 特性三。

## 快速开始

` ` `bash
npm install my-package
` ` `

` ` `ts
import { doSomething } from 'my-package';
doSomething();
` ` `

## 文档

完整文档见 [docs/](./docs/)。

## 许可

MIT © 2026 ……
```

(实际写作时三反引号不需要空格)

**纪律**:

- **不要**堆砌徽章(Badge)塞满 README 头部,最多 2–3 个最关键的(CI、版本、license)。
- **快速开始**必须在 README 第一屏可以直接复制粘贴运行。
- **示例代码**必须是**真实可跑**的最小完整示例,不要省略 import。

### API / 接口文档

- 每个接口:**动词 + 路径**作为三级标题,下面分"请求""响应""错误码""示例"四段。
- **严禁**只写参数表不给示例。
- JSON 示例使用真实字段和示意值,不要用 `string` / `number` 作为字段值。

## 六、商务文案(提案、方案、介绍稿)

### 写作心法

- **换位**:永远从读者视角写。读者是投资人,讲回报和风险;读者是客户,讲价值和落地;读者是合伙人,讲愿景和分工。
- **一页纸**:所有提案都应该能浓缩到"一页纸摘要",把核心观点、关键数字、下一步动作写清。
- **先讲好处,再讲方法**:不要开头就是技术架构图。先讲"这会让你赚到什么/省到什么",再讲"我们怎么做"。

### 基础骨架

```markdown
# [方案名]

> **给谁看**:XX 部门 / 某客户 / 投资人
> **一句话**:用 XX 方式帮 XX 实现 XX,预计 XX。

## 1. 我们看到的问题

(不超过 300 字,直指痛点,最好引用一两个具体场景或数据。)

## 2. 我们的解法

(核心思路,可以配一张 `mindmap` 或 `flowchart` 讲清楚结构。)

## 3. 为什么能成

- **差异化 1**:…… (和市面上的 XX 有什么不一样)
- **差异化 2**:…… (我们独有的什么资源/技术/认知)
- **差异化 3**:……

## 4. 如何落地

阶段一(第 X 周—第 Y 周):……
阶段二(第 Y 周—第 Z 周):……
……

## 5. 投入与回报

| 项目 | 投入 | 预期回报 |
|------|------|----------|
| …… | …… | …… |

## 6. 下一步

一句话:需要对方做什么决策 / 提供什么配合。
```

## 七、产出前的自检清单

交付之前,过一遍:

- [ ] **体裁判断对了吗?** 这是文学体还是信息体?排版方式和体裁匹配吗?
- [ ] **第一屏有没有"废话开头"?** 有没有不先讲结论、先讲"近年来/随着"的毛病?
- [ ] **段落是段落吗?** 有没有明明一句话能说清的事被拆成了 3 条列表?
- [ ] **加粗和强调是不是太多了?** 能不能删掉一半?
- [ ] **标题是不是过密?** 每个二级标题下是不是都有至少一个完整段落?
- [ ] **charts 是否滥用?** 文学体里有没有混进去图表?信息体里的图表是否每一个都真的帮到了理解?
- [ ] **emoji/装饰字符是否过度?** 是否符合这个体裁的"严肃度"?
- [ ] **如果你是读者,愿意从头读到尾吗?**

最后一条是最重要的。如果自己都不想读,别交付。

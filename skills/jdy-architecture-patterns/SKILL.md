# 简道云架构范式判定法

## 五步判定流程
1. **本体论检验**: 判断业务实体核心价值是"记录存在"、"流程流转"、"关系网络"还是"定制展示"。
2. **复杂度分布**: 识别 60% 以上的业务复杂度落在哪一维。
3. **信号检测**: 检查是否出现"自动化爆炸"或"链式触发脆弱"。
4. **AI 能力匹配**（🆕 V10.16.0+）: 检查需求是否可通过 AI 推荐字段、AI 填充、AI 执行节点等零代码实现，避免不必要的二开。
5. **ROI 检验**: 估算简道云标准功能的覆盖度（含自定义详情页、集成应用、智能助手 Pro 等新能力）。

## 表关系设计规范
- **关联方式**: 使用"关联表单"控件，严禁使用数据库外键术语。
- **待办同步**: 明确主表与待办表之间的数据同步逻辑（智能助手/API）。
- **颜色标识**: 基础档案（蓝）、业务主表（绿）、明细表（灰）、待办表（橙）。

## 架构范式清单

### 范式 A：标准表单范式
- **核心特征**: 业务价值 = "记录存在"，CRUD 为主
- **实现层**: L0-L1（表单 + 仪表盘）

### 范式 B：流程驱动范式
- **核心特征**: 业务价值 = "流程流转"，审批/会签/分支为主
- **实现层**: L0-L2（流程表单 + 智能助手）

### 范式 C：自定义详情页范式（🆕 V10.10.0 上线，V11.2.0 成熟）
- **核心特征**: 业务价值 = "定制展示"，需要在简道云内部构建高度定制化的数据展示页面
- **能力矩阵**: 图表组件、列表组件、简报卡片、按钮交互、打印模板、组件显隐规则、移动端支持、多数据源
- **实现层**: L0-L2（自定义详情页零代码配置）
- **替代场景**: 原本需要 iframe 嵌入或前端自定义页面的展示需求，现在可通过自定义详情页零代码实现
- **人天基准**: 2.0-4.0 人天/页（介于仪表盘和前端自定义页面之间）

### 范式 D：集成应用范式（🆕 V11.0.0 上线）
- **核心特征**: 业务价值 = "系统集成"，需要与外部系统深度对接
- **实现层**: L2-L3（集成应用 + 智能助手 Pro）
- **注意**: 能力边界仍在演进中，评估时需确认最新功能范围

### 范式 E：AI 增强范式（🆕 V10.16.0 AI 实验室上线）
- **核心特征**: 业务价值 = "智能处理"，需要 AI 辅助决策/填充/分类
- **能力矩阵**: AI 推荐字段、AI 填充、AI 编辑助手、AI 执行节点、AI 生成公式、AI 助理
- **实现层**: L0-L2（AI 功能零代码配置）
- **替代场景**: 原本需要二开的智能分类、自动填充、数据分析场景

### 范式 F：双模式协同范式（🆕 2026-06-14）
- **核心特征**: 业务价值 = "全局聚合 + 单条深度交互"，二开页面做"面"，简道云原生页面做"点"
- **设计原则**:
  - **模式 A（数据汇聚页）**: 二开定制，负责跨多条数据的聚合展示（甘特图、资源日历、看板）和复杂计算可视化（CPM、冲突检测）
  - **模式 B（原生详情页）**: 简道云原生，负责单条记录的字段编辑（CRUD）、流程审批、智能助手触发、按钮操作
  - **链接纽带**: 模式 A 中每条数据旁嵌入链接，点击跳转到模式 B 的简道云原生详情页
  - **数据闭环**: 模式 B 编辑 → 智能助手触发 → Webhook 回调 Bridge 层 → 重新计算 → 写回简道云 → 模式 A 刷新可见
- **适用判定**: 项目同时需要"全局聚合视图"和"单条记录深度交互"，且两者需要数据联动
- **实现层**: L2-L4（二开页面 + 简道云原生 + Bridge 后端）
- **人天基准**: 模式 A 页面按前端定制估算 + 模式 B 页面 0 人天（原生能力）+ 链接嵌入 0.5 人天/页
- **关键约束**:
  - 简道云表单**不只是数据库**——它同时也是前端交互闭环的一部分
  - 模式 A 页面**禁止**自行实现 CRUD 表单编辑——必须链接到简道云原生详情页完成
  - 模式 B 页面的数据变更**必须**通过智能助手/Webhook 通知 Bridge 层，确保汇聚页数据一致
- **典型案例**: 项目计划管控系统（甘特图=模式A，任务详情=模式B）、资源调度系统（资源日历=模式A，资源详情=模式B）

> ⚠️ **与范式 C（自定义详情页）的区别**: 范式 C 是简道云内部的零代码定制展示，不涉及二开页面。范式 F 是二开页面与简道云原生页面的**跨系统协同**。当自定义详情页无法满足展示需求（如甘特图、网络图）时，升级为范式 F。

## 混合架构边界（🔄 已更新）
- **L0-L2**: 简道云原生实现（表单/流程/仪表盘/**自定义详情页**/**集成应用**/**AI 功能**）。
- **L3**: 智能助手 Pro（含 HTTP 请求/Agent/循环容器/AI 大模型节点）+ 少量二开（Webhook/自定义页面）。
- **L4**: 完整后端服务（OpenAPI 聚合/复杂计算/外部系统集成）。

> ⚠️ **重要变化**：智能助手 Pro 的能力已大幅增强（HTTP 请求节点 V10.9.0、Agent 节点 V10.19.0、循环容器 V9.15.0、AI 大模型节点 V11.0.0），很多原本需要 L3/L4 二开的自动化场景现在可通过 L3 零代码实现。评估时应优先评估智能助手 Pro 的覆盖度，再决定是否需要二开。

## 各范式权限隔离检查项（🆕）

> 判定架构范式后，必须同步评估该范式的权限风险。详见 `jdy-permission-model` 技能。

| 范式 | 权限状态 | 必检项 |
|------|---------|--------|
| **A：标准表单** | ✅ 原生完整 | 确认数据范围规则（本人/本部门/全部）已配置 |
| **B：流程驱动** | ✅ 原生完整 | 确认流程节点负责人与审批权限匹配 |
| **C：自定义详情页** | ⚠️ 基本完整 | 检查是否有按钮跳转外部页面（需身份透传） |
| **D：集成应用** | ⚠️ 部分断裂 | 确认集成应用层的数据过滤方案 |
| **E：AI 增强** | ✅ 原生完整 | 确认 AI 功能的数据访问范围 |
| **L3：智能助手 Pro** | ⚠️ 部分断裂 | HTTP 请求节点需在 body/header 中传递用户身份 |
| **F：双模式协同** | ⚠️ 部分断裂 | 模式 A 页面需 OAuth2.0 身份透传；模式 B 页面权限原生完整；检查链接跳转时身份是否丢失 |
| **L4：完整后端** | ❌ 完全断裂 | **必须设计三层权限重建方案**（身份/数据/操作），预算 3-5 人天 |

### L4 二开场景权限铁律

1. **API 查询必须带 filter**：不能裸查全表（API Key 是上帝视角，绕过原生权限）
2. **iframe 必须走 OAuth2.0**：不能用 URL 参数传递身份
3. **创建操作必须传 data_creator**：否则数据归属变成管理员
4. **流程操作必须验证 username 与 task_id 对应**

## 关联技能（🔄 已更新）
- **jdy-custom-dev-capabilities**: 判定 L3/L4 层二开方案时，必须用该技能检查平台硬性约束（API 限制、插件沙盒、子表单行数等）。自定义详情页可替代部分 iframe 场景时，用该技能确认前端可定制范围。
- **cost-estimation-model**: 架构范式确定后，用成本模型估算二开人天。自定义详情页（2-4 人天/页）和 AI 功能配置的人天基准已更新。**L4 场景需额外估算权限重建人天（3-5 人天）**。
- **jdy-permission-model**（🆕）: 判定范式后，用该技能的"断裂点矩阵"评估权限风险。L4 场景必须用"三层权限重建架构"设计方案。

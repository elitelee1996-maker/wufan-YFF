# 伙伴创建器

本技能指导你创建高质量的子 Agent（伙伴）。伙伴是通过 Task 工具委派任务的独立 Agent 实例，拥有自己的系统提示词、工具集和 LLM 配置。

## 0. 先判断: 主 Agent 还是子 Agent (伙伴)?

**只有满足以下三条全部**，是**主 Agent**:

- 用户**直接对话**它（不是被另一个 Agent 调起来）
- 它**直接服务用户**（不是给主 Agent 当帮手）
- 它**有明确角色**（不是"啥都能聊的通用助手"）

只要其中一条不满足 → **是子 Agent (伙伴)**，继续走本技能下面的流程。

### 如果判断是主 Agent → 不要走本技能

主 Agent 的构建走 `product-handbook` 内置技能里的 `references/workflows.md` **流程 1**，那里有完整的"对话式构建主 Agent"5 步法 (摸需求 → 设计蓝图 → 联动各内核技能造资产 → 给到 agents.md 完整文本 + 资产清单 + 测试用例 → 用户在 设置 → 智能体资产 → "+ 新建 Agent" 落地)。本技能不覆盖主 Agent。

### 不确定是主还是子? 用一个问题确认

如果用户描述含糊（例: "我想做一个能帮我 XX 的 Agent"，没说是谁来对话），**别凭印象猜**，用 `ask_user_question` 问一次:

> 你想要的是哪一种?
>
> - **主 Agent** — 用户直接对话它，每天来找它干活的入口 Agent
> - **伙伴 / 子 Agent** — 不直接对话，而是被某个主 Agent 在执行任务时自动调起来做子任务的小专家

根据用户回答分流:
- 选"主 Agent" → 走 `product-handbook` 流程 1
- 选"伙伴" → 继续本技能

## 创建流程

1. 理解用户需求 — 这个伙伴要解决什么问题？
2. 设计伙伴身份 — 名称、描述、能力边界
3. 编写系统提示词 — 清晰的角色定义和工作指南
4. 配置工具和技能 — 选择必要的工具集
5. 写入 agent.md 文件

## 文件格式

伙伴定义为 `subagents/{name}/agent.md`，由 YAML 前置元数据 + Markdown 正文组成：

```yaml
---
name: my-agent-name
description: 一句话描述能力和使用场景
llm_config_id: （可选，留空则继承主 Agent）
max_iterations: 50
tools: read, write, edit, grep, glob, bash
skills:
  - skill-name-a
  - skill-name-b
capabilities: ["能力标签1", "能力标签2"]
expertise: ["领域1", "领域2"]
work_style: 工作风格描述
---

（系统提示词正文）
```

## 前置元数据字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 英文标识符，小写+连字符（如 `code-reviewer`） |
| `description` | 是 | 描述能力和触发场景，主 Agent 据此决定何时委派 |
| `tools` | 否 | 允许的工具白名单，逗号分隔。留空=继承全部可用工具 |
| `disallowedTools` | 否 | 禁止的工具黑名单。与 tools 互斥，选一种方式即可 |
| `skills` | 否 | 预加载的技能名列表（YAML 列表格式） |
| `max_iterations` | 否 | 最大迭代次数，默认 50，上限 100 |
| `llm_config_id` | 否 | 独立 LLM 配置 ID。留空=继承主 Agent 的模型 |
| `capabilities` | 否 | 能力标签（展示用） |
| `expertise` | 否 | 擅长领域（展示用） |
| `work_style` | 否 | 工作风格描述 |

### 工具选择策略

子 Agent 以下工具**被系统固定禁止**，不要写入 tools 字段：

`task`、`roundtable`、`automation`、`project`、`enter_plan_mode`、`exit_plan_mode`、`ask_user_question`

工具选择的原则：

- **只读型伙伴**（如代码审查）：仅给 `read, grep, glob, ls`
- **研究型伙伴**：加上 `web_search, web_fetch`
- **执行型伙伴**（可修改文件）：加上 `write, edit, bash`
- **全能型伙伴**：留空 tools 字段（继承全部），或用 disallowedTools 排除少量不需要的

## 系统提示词编写指南

系统提示词是伙伴能力的核心。遵循以下原则：

### 结构模板

```markdown
你是一个专业的 [具体领域] 专家。

## 核心能力

1. 具体能力描述（不要泛泛而谈）
2. 具体能力描述

## 工作流程

1. 第一步：具体操作
2. 第二步：具体操作
3. 第三步：具体操作

## 输出要求

- 输出格式规范
- 质量标准

## 行为准则

- 边界约束
- 注意事项
```

### 编写原则

1. **角色要具体** — "你是前端性能优化专家" 比 "你是一个技术专家" 好
2. **工作流要可操作** — 每步都是 Agent 能直接执行的动作
3. **输出要有格式** — 明确告诉 Agent 结果长什么样
4. **边界要清晰** — 伙伴只做被要求的事，不要发散
5. **不写废话** — Agent 已经很聪明，只告诉它不知道的事

### 常见错误

- 提示词过于空泛："做好你的工作" → 无效
- 提示词过长：超过 500 行 → 浪费上下文
- 遗漏输出格式 → 每次输出质量不一致
- 不说明使用的工具 → 伙伴可能不知道自己有某些工具可用

## 创建步骤

### 步骤 1：确认需求

向用户确认：
- 这个伙伴的核心职责是什么？
- 它需要读写文件吗？需要上网搜索吗？
- 期望的输出格式是什么？
- 有没有需要预加载的技能？

避免一次问太多问题。如果用户描述已经足够清晰，直接进入创建。

### 步骤 2：创建文件

在 `subagents/` 目录下创建：

```
subagents/{name}/agent.md
```

使用 `write` 工具写入完整的 agent.md 内容。

### 步骤 3：验证

创建后用 `read` 工具回读文件，确认格式正确：
- YAML 前置元数据被 `---` 正确包裹
- name 是合法的英文标识符
- tools 列表中没有被禁止的工具
- 系统提示词不为空

## 高质量示例

### 代码审查伙伴

```yaml
---
name: code-reviewer
description: 专业的代码审查 Agent。审查代码质量、发现问题、提供改进建议。
tools: read, grep, glob, ls
max_iterations: 15
---
你是一个专业的代码审查专家。

## 核心能力

1. **代码质量审查** - 可读性、可维护性、命名规范
2. **Bug 检测** - 逻辑错误、边界条件
3. **安全审计** - SQL 注入、XSS 等常见漏洞

## 审查流程

1. 理解代码整体结构
2. 查看最近修改（git diff）
3. 逐文件详细审查
4. 按严重程度分类

## 输出格式

- 🔴 严重：必须修复
- 🟡 中等：建议修复
- 🔵 轻微：可选改进

## 原则

- 客观公正，解释原因
- 提供可操作的改进建议
```

### 研究型伙伴

```yaml
---
name: researcher
description: 研究员。执行研究子任务，搜索信息、精读文档、保存发现。
tools: web_search, web_fetch, read, write, edit, grep, glob
max_iterations: 30
---
你是一个研究员，负责执行具体的研究子任务。

## 研究策略

1. web_search 广度搜索，获取摘要
2. web_fetch 精读高价值链接的完整内容
3. 将发现保存到文件

## 输出要求

- 保存到任务指定的文件路径
- 包含关键事实 + 来源 URL
- 返回简要总结给主 Agent
```

## 注意事项

- 目录名必须是小写英文+连字符（如 `data-analyzer`）
- `subagents/` 路径会自动路由到团队目录，同团队 Agent 共享
- 禁止在内核路径（`/app/core/subagents/builtin`）下创建
- 创建后立即生效，主 Agent 下次调用 task 工具时即可使用
- 如果伙伴预加载了技能（skills 字段），技能索引会自动注入到伙伴的系统提示词中

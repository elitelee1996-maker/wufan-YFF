## Top 20 高频风险清单

### 1. 对接人时间不足或态度不积极，沟通困难
- **出现频次**: 288次
- **风险分类**: 人员
- **影响程度**: 高
- **应对建议**:
  - 项目启动时明确对接人职责
  - 建立定期沟通机制（周例会）
  - 要求甲方高层支持，确保对接人投入时间
  - 设置沟通响应时效条款

### 2. 甲方对项目进度要求较高，需在XX时间内上线
- **出现频次**: 271次
- **风险分类**: 时间
- **影响程度**: 高
- **应对建议**:
  - 充分评估工期可行性
  - 明确需求范围，避免范围蔓延
  - 设置里程碑节点，分阶段交付
  - 预留缓冲时间（建议15-20%）
  - 明确延期责任条款

### 3. 二开工作量占比超出搭建工作量，后期运维成本高
- **出现频次**: 188次
- **风险分类**: 范围
- **影响程度**: 高
- **应对建议**:

  - 明确二开范围和边界
  - 评估二开功能的可维护性
  - 单独报价二开运维费用
  - 引导客户接受标准功能

### 4. 需求变更（对接人权力不足、跨部门需求多、二开需求不明确等）

| 风险等级 | 出现频次 | 应对策略 |
|---------|---------|---------|

| **P1 高** | 100-200次 | 需在合同中明确条款，预留缓冲 |
| **P2 中** | 50-100次 | 需在项目计划中考虑，预留时间 |

## 行业特定风险模式

### 制造业特定风险
- 生产流程复杂，需求蔓延风险高
- BOM表管理复杂，数据量大
- 多部门协作，权限设计复杂
- 二开需求多（如MES集成）

### 建筑业特定风险
- 项目周期长，进度管理复杂
- 多项目并行，资源协调困难
- 现场环境复杂，移动端需求高
- 分包管理复杂

### 教育行业特定风险
- 预算有限，报价敏感
- 决策周期长，涉及多部门
- 招生季节性强，上线时间要求高
- 数据隐私要求高

### IT/软件行业特定风险
- 客户IT背景强，需求明确但复杂
- 集成需求多（如Git、Jenkins）
- 对系统性能要求高
- 二开需求多

## 风险识别检查清单

项目启动前，逐项检查以下风险：

### 人员风险
- [ ] 对接人是否有足够时间和权力？
- [ ] 是否涉及多部门，存在理解偏差风险？
- [ ] 甲方客情关系是否良好？
- [ ] 最终用户是否愿意接受新系统？

### 范围风险
- [ ] 需求是否明确，变更风险是否可控？
- [ ] 二开工作量是否超出搭建工作量？
- [ ] 是否涉及多厂商配合，权责是否清晰？
- [ ] 自动化要求是否过高？

### 系统风险
- [ ] 简道云版本功能是否满足需求？
- [ ] 数据量是否超出系统上限？
- [ ] 组织架构是否复杂，权限设计是否困难？
- [ ] 用户操作环境是否支持？

### 时间风险
- [ ] 甲方进度要求是否过高？
- [ ] 是否有明确的上线时间节点？
- [ ] 是否预留了缓冲时间？

### 流程风险
- [ ] 验收流程是否清晰？
- [ ] 回款周期是否可接受？
- [ ] 是否有监理机构介入？

### 团队风险
- [ ] 甲方信息化程度是否足够？
- [ ] 是否有IT团队支持？
- [ ] 是否有外部团队配合？

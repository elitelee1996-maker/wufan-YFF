# 业务需求描述写法经验

基于620个报价单中2,994条实际业务需求描述的写法模式分析，总结出以下经验。

## 一、总体发现

| 维度 | 数据 |
|------|------|
| 分析样本 | 2,994条有效需求描述 |
| 短描述（30-80字） | 80.9% — 绝大多数是一句话概括 |
| 中描述（80-150字） | 13.4% |
| 长描述（150字+） | 5.7% |
| 好描述（有字段+角色+逻辑） | 13.5% |
| 中等描述 | 61.6% |
| 差描述（太笼统/无内容） | 24.9% |

**核心结论**：近1/4的需求描述质量不达标。好描述的共同特征是"具体、可执行、可验收"。

## 二、6种写法公式

### 公式1：一句话概括式（最常见，占80%）

**结构**：`[动作] + [对象] + [字段列举]`

**适用场景**：普通表单、基础数据表

**好的写法**：
```
记录存货类型（原材料/在产品/产成品），产品编码、产品名称、规格型号、检验标准信息
```
```
记录员工个人信息、学历信息、银行卡信息、紧急联系人、家庭信息、个人材料，关联出个人的招聘需求，入职表
```
```
记录设备编号、M/C、名称、规格型号、数量、完好状态、使用场所、生产厂、生产日期、置购日期、维护频次
```

**要点**：
- 必须列举具体字段（用顿号分隔）
- 字段数量3-10个为宜
- 如有关联表，写明"关联出XXX"

**差的写法**（对比）：
```
❌ 登记甲方信息，可以直接导入系统已有的相关信息
```
→ 没有列举具体字段，"相关信息"太模糊

---

### 公式2：编号列表式（高质量描述常用）

**结构**：`总述（1-2句） + 1.xxx 2.xxx 3.xxx`

**适用场景**：流程表单、复杂业务模块

**好的写法**：
```
系统提供库内管理，实现库管员对库内出入业务、盘点业务、调拨业务等进行管控，
同时能提供各种库内统计报表供库房人员进行数据分析，具体需求如下:
1) 满足库房日常出入库、仓内调拨、盘点、物料转换、组装拆卸等日常业务开展。
2) 支持多计量单位、批次、仓库、库位、有效期、货主等多属性多维度仓内管控。
3) 提供多维度的数据分析，如:库存台账、账龄分析、收发存汇总表、呆滞料分析。
4) 发货管理主要实现待发货订单的跟踪管理，支持发货单的开单新增、修改、删除、送审、跟踪发货收货管理。
```

**要点**：
- 总述说清楚业务价值和整体目标
- 每个编号项是一个独立的功能点
- 每项用"满足/支持/提供/实现"开头
- 列举具体的业务场景或字段

---

### 公式3：角色-动作-结果式

**结构**：`[谁] + [做什么] + [系统如何处理] + [达到什么效果]`

**适用场景**：审批流程、多人协作场景

**好的写法**：
```
销售业务员将订单/合同信息录入系统管理。用户可根据此数据进行留档、分析。
并以此作为仓储出库的依据。
```
```
每个工作日生成一条员工应出勤数据，程序未匹配到打卡、补卡、请假数据即判定为旷工，
新增一条旷工数据，同时触发提醒。后期补卡，需要触发程序删除相应旷工数据。
```
```
检测分析人员将分析结果录入系统管理。用户可根据此数据进行产品在制及实际产出分析。
生产检验支持实时跟踪产线信息，推送消息给质检人员、实时完成检后线上上报检验数据。
```

**要点**：
- 明确操作角色（谁来做）
- 明确系统行为（自动做什么）
- 明确触发条件（什么情况下）
- 明确结果（达到什么效果）

---

### 公式4：场景-条件-动作式

**结构**：`当/如果[条件] → 则[动作] → [结果]`

**适用场景**：有分支逻辑的业务流程

**好的写法**：
```
销售人员根据客户需求，下达销售订单，可勾选对应的预售订单编号，带出客户信息。
销售订单须明确付款方式、关键的收款节点、销售明细等情况，
并会在关键收款节点前三到五天进行推送提醒，提醒销售进行收款动作，
若款已收则不会触发提醒。
```
```
仓库根据到货情况，进行收货登记，并关联好采购订单编号。
仓库部门需针对货物进行质检，并将货物发至不同的仓库当中去，
若发现货物存在损坏等问题，则将该货物入维修仓中，后续进行维修处理。
```
```
扫描设备二维码进行点检和保养流程，异常触发报修子流程（标记设备状态"异常"，
报修流程结束后恢复"正常"）。
如果最终定义为计划维修项的话，则此项添加进该设备的保养项目池，
在之后保养单的建立时供选择。
```

**要点**：
- 先描述正常流程
- 再用"若/如果/当"描述分支条件
- 每个分支都要说清楚处理方式和结果

---

### 公式5：数据流式（对接/集成场景）

**结构**：`[数据源] → [处理方式] → [目标系统] + [字段说明]`

**适用场景**：API对接、数据同步、系统集成

**好的写法**：
```
初步需求是同步金蝶中的基础信息，并和简道云中的数据进行计算，
并且也有简道云审批通过的数据返回给金蝶，
故具体接口数量不定，暂时以每个接口2-3人天为评估。
```
```
ERP数据同步（日期、单据号、产品编号、数量、单价、金额、交货日期）
```
```
用户录入次品信息，将用户次品信息推送至pivot88系统中。
pivot88系统的权限关系如本文档"Data Mapping Guidelines"所示。
数据对接包含新增和修改的同步。
```

**要点**：
- 明确数据流向（从哪到哪）
- 列举同步的具体字段
- 说明同步方式（实时/定时/手动）
- 说明同步类型（新增/修改/删除）

---

### 公式6：业务价值+功能拆解式（最佳写法）

**结构**：`[业务背景/价值] + [功能编号列表] + [具体字段/规则]`

**适用场景**：核心业务模块、需要详细评估的功能

**好的写法**：
```
系统提供销售合同信息录入，销售业务员将订单/合同信息录入系统管理。
用户可根据此数据进行留档、分析。并以此作为仓储出库的依据。
通过订单管理系统平台，满足企业从销售部门接单、生产部门排定计划、
机加车间加工，质检班各阶段检验，仓储部门仓内管理一体化业务流程贯通。
消除各部门信息孤岛，提高内部运营效率。

1) 在销售部门业务员接到客户需求时，满足业务员对客户需求的快速录入，
   包括客户名称、产品名称、产品规格尺寸、订单数量、交货日期、材质要求、包装要求等信息。
2) 满足业务员可以通过销售订单及时获取下游的生产订单进度、入库质检数量及计划完工时间等信息。
3) 满足业务员在销售订单环节可以查询库存情况。
```

**要点**：
- 开头1-2句说清楚业务价值（为什么要做）
- 编号列表拆解具体功能点（做什么）
- 每个功能点列举具体字段或规则（怎么做）
- 结尾可补充扩展性说明

## 四、常见写法错误（必须避免）

### 错误1：太笼统，无具体内容
```
❌ 采购根据车间汇报材料内容进行清单申报，后续可导出excel即可
❌ 将订单信息传至简道云表单，并可建立看板，便于掌握订单信息情况
```
**问题**：没有说清楚具体字段、业务规则、操作角色
**修正**：列举具体字段，说明业务规则

### 错误2：功能名和描述完全一样
```
❌ 功能名: 质量管理
   描述: 质量管理
```
**问题**：描述没有提供任何额外信息
**修正**：描述应该说明"谁在什么场景下做什么，系统如何处理"

### 错误3：只有结果，没有过程
```
❌ 后续可导出excel即可
❌ 生成对应要求的EXCEL文件，可下载
```
**问题**：只说了最终输出，没有说清楚数据来源、处理逻辑
**修正**：说明数据从哪来、经过什么处理、输出什么格式

### 错误4：写的是调研问题而非需求
```
❌ 是否有老系统？当前项目需求与老系统业务什么关系？
❌ 甲方内部做项目验收是否需要走特定流程？
```
**问题**：这是调研阶段的问题，不是需求描述
**修正**：调研完成后，将答案转化为具体的需求描述

### 错误5：描述中混入内部评审信息
```
❌ 备注：蓝色标注列为内部评审需要，发给客户前需自行删除
```
**问题**：内部信息泄露到客户版本
**修正**：使用双版本机制，内部版和客户版分开

## 五、按功能类型的写法模板

### 普通表单
```
记录[业务对象]的[核心属性1]、[核心属性2]、[核心属性3]等信息，
关联出[关联表名]，支持[导入/导出/打印]。
```

### 流程表单
```
[角色]发起[业务名称]流程，填写[关键字段1]、[关键字段2]等信息，
提交后由[审批角色]审批。审批通过后[系统自动做什么]。
若[条件]，则[分支处理]。

1) 满足[场景1]的需求，包括[字段列举]
2) 支持[场景2]的处理
3) 审批通过后自动[更新/推送/生成]XXX
```

### 仪表盘
```
通过[数据来源]的数据，进行[处理方式]，
以[图表类型1]、[图表类型2]等形式展示[分析维度]，
支持[筛选/下钻/联动]。
```

### 数据对接
```
从[源系统]同步[数据类型]到简道云，
同步字段包括：[字段1]、[字段2]、[字段3]。
同步方式：[实时/定时/手动]。
同步类型：[新增/修改/删除]。
[如有反向同步]简道云审批通过的数据返回给[目标系统]。
```

### 代码开发
```
[业务场景描述]。
技术方案：[简述实现方式]。
输入：[输入数据/参数]。
输出：[输出结果]。
约束：[性能/兼容性/安全要求]。
```

## 六、质量自检清单

写完需求描述后，逐项检查：

- [ ] 是否列举了具体字段？（不是"相关信息"）
- [ ] 是否明确了操作角色？（谁来做）
- [ ] 是否说明了业务规则？（什么条件下做什么）
- [ ] 是否描述了系统行为？（自动做什么）
- [ ] 是否可以据此验收？（能不能判断做没做完）
- [ ] 是否可以据此估算人天？（工作量是否可评估）
- [ ] 是否去除了内部评审信息？（客户版本是否干净）

**通过6项以上 = 好描述**
**通过3-5项 = 中等描述**
**通过2项以下 = 差描述，需要重写**

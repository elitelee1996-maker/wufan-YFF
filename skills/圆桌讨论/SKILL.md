# 圆桌频道设计指南

## 决策：做 vs 讨论 vs 调研+讨论

发起讨论前先做判断：

| 情况 | 行动 |
|------|------|
| 任务明确，一个人能做好 | 直接做或用 `task` 委派，不需要讨论 |
| 需要多视角碰撞，且你已掌握足够信息 | 直接发起 `roundtable` |
| 讨论依赖你不确定的事实/数据/最新信息 | **先用 `task` 调研，再发起讨论** |

### "先调研再讨论" 工作流

当讨论主题超出 LLM 知识（如市场动态、竞品分析、最新技术），必须先收集信息：

```
1. task(tasks="调研 X 的最新市场格局和关键数据", subagent="网络研究员")
2. 拿到调研结果后，将关键发现写入 roundtable 的 topic 中
3. roundtable(action="create", config_json='...')
   → topic 中包含调研得到的事实基线
```

这确保参与者在有充分信息的基础上讨论，而非空谈。

## Topic 表述原则

topic 是所有参与者看到的第一句话，直接决定讨论质量。

- **忠实还原用户意图**：不要替用户"优化"表述，保持原始问题的核心
- **附加上下文**：如果做了前置调研，把关键发现摘要附在 topic 中
- **明确期望产出**：在 topic 末尾写清"请分析 X 并给出 Y"

坏例：`topic: "OpenAI后的ToB AI机会"`（丢失了用户原意）

好例：`topic: "在 OpenAI 开放大模型 API 之后，ToB 领域还有哪些差异化的 AI 产品机会？请结合以下调研发现分析：[摘要]。期望产出：3-5 个具体方向 + 可行性评估"`

## 迭代深度指导

圆桌参与者拥有完整的工具能力（搜索、文件读写等），和 `task` 工具共享同一套伙伴生态。
参与者在多轮讨论中保持完整跨轮记忆——包含自己的思考过程、工具调用结果和发言内容。
通过 `max_iterations` 控制每个参与者每次发言的调研深度：

| 场景 | max_iterations 建议 |
|------|-------------------|
| 需要查证事实的讨论（市场分析、技术调研） | 为关键参与者设 6-8 |
| 纯观点碰撞（辩论、头脑风暴） | 保持默认或设 2-3 |
| 混合场景 | "调研者"角色设高值，"评论者"角色设低值 |

示例：
```json
"agents": [
  {"kind": "subagent", "ref_name": "网络研究员", "perspective": "...", "max_iterations": 8},
  {"kind": "temp", "display_name": "评论家", "perspective": "...", "max_iterations": 3}
]
```

不设 `max_iterations` 时的默认值：main=8, subagent=6, derived=5, temp=4。

## 模式选择

| 模式 | 选择条件 |
|------|---------|
| `moderator` | 需要结构化推进、有明确评审标准、需要有人把控节奏 |
| `ordered` | 每人各抒己见即可，不需要复杂互动 |
| `free` | 创意发散、辩论、需要观点碰撞和追问 |

## 参与者设计

核心原则：

1. **数量克制** — 3-4 个参与者效果最佳，超过 5 个容易分散
2. **视角正交** — 每个参与者的 perspective 必须提供不同维度的价值
3. **perspective 要具体** — 写清"你要关注什么、忽略什么、以什么标准判断"
4. **优先使用真实伙伴** — 主 Agent 和子代理有专业的模型配置和工具，比临时角色更强

好的 perspective：
```
"从技术实现成本角度分析。关注：开发周期、技术风险、人力需求。如果成本过高，直接指出并建议替代方案"
```

差的 perspective：
```
"从技术角度分析"
```

## 发起讨论后的行为

调用 `roundtable(action="create")` 成功后，讨论在后台完全异步运行，通常需要数分钟。

讨论完成后系统会**自动推送**结论给你——你不需要主动查询。
因此不要调用 `roundtable(action="status")` 反复轮询进度，这只会浪费迭代次数。

你有两种选择：
1. 如果用户的任务到此结束（仅仅是发起讨论），告知用户"讨论已发起"后结束即可
2. 如果用户的任务还有其他部分（圆桌只是其中一环），可以继续执行其他工作

讨论完成后系统会自动推送结论和记录文件路径，届时你再根据用户的完整任务需求决定后续行动。

## 讨论结束后的处理

频道结束后，引擎会自动完成：

1. **独立总结者生成结论** — 从你（主 Agent）派生一个不参与讨论的总结者，客观汇总各方观点、共识、分歧和行动建议
2. **讨论记录持久化** — 完整讨论过程保存为 `圆桌讨论-{名称}.md` 文件到会话文件目录
3. **轻量化注入你的上下文** — 你收到的是结论摘要 + 文件路径引用（而非全量讨论文本）
4. **自动触发你运行** — 你会被触发运行一轮，向用户汇报讨论结果

你在汇报时应该：
1. 向用户简要汇报讨论结果（基于注入的结论摘要）
2. 提取可执行的行动项
3. 如果需要查看讨论细节，用 `read_file` 读取讨论记录文件
4. 如果需要基于结论执行具体工作，用 `task` 委派

## 使用已有伙伴发起讨论

当用户说"用已有的伙伴/团队来讨论"时，使用 `kind: "main"` 或 `kind: "subagent"` 引用伙伴而非创建临时角色。

**关键规则**：
- `kind: "main"` — 引用主 Agent，`ref_id` 可以填名称或 UUID（推荐用名称，更直观）
- `kind: "subagent"` — 引用子代理，`ref_name` 填子代理名称（可用伙伴列表中的 name 字段）
- 引用伙伴时仍可附加 `perspective` 和 `role` 来引导其讨论方向
- 使用伙伴列表中的真实名称，不要编造

**示例**：用户有子代理 `researcher` 和 `coder`，要讨论技术方案：
```json
{
  "display_name": "技术方案讨论",
  "topic": "评估微服务拆分方案的利弊",
  "speaking_mode": "free",
  "max_rounds": 4,
  "agents": [
    {"kind": "subagent", "ref_name": "网络研究员", "perspective": "调研业界实践并提供数据支持", "max_iterations": 8},
    {"kind": "subagent", "ref_name": "coder", "perspective": "从代码实现和工程复杂度角度评估"},
    {"kind": "temp", "display_name": "架构师", "perspective": "从系统整体架构和长期可维护性角度"}
  ]
}
```

**示例**：混合引用主 Agent + 临时角色：
```json
{
  "display_name": "产品方向辩论",
  "topic": "下季度应优先做 A 还是 B？",
  "speaking_mode": "moderator",
  "max_rounds": 5,
  "agents": [
    {"kind": "temp", "display_name": "主持人", "role": "moderator", "perspective": "中立主持讨论"},
    {"kind": "main", "ref_id": "Agent名称", "perspective": "全局决策视角"},
    {"kind": "temp", "display_name": "魔鬼代言人", "perspective": "刻意反对，挑战主流意见"}
  ]
}
```

## 调用注意事项

**最常见的失败原因**是 `config_json` 参数传递方式不正确：
- **正确**：`config_json` 作为顶层参数直接传入
- **错误**：把 `config_json` 嵌套到其他参数中（如放进了 `kwargs`）
- 如果工具返回"至少需要 2 个参与者"而你确定传了 agents，说明参数结构有误，检查 config_json 是否正确传递

## 反面模式（避免）

- **反复轮询状态** — create 成功后反复调用 `roundtable(action="status")` 查进度，浪费迭代次数且无法加速。系统会在讨论完成后自动推送结论。
- 让 6 个临时角色空谈没有数据支撑的话题
- 把用户的 topic 大幅改写导致偏离原意
- 所有参与者 perspective 类似（如都是"从 XX 角度分析"）
- 需要查证的事实性话题不先调研
- 所有参与者 max_iterations 都设很高（浪费资源，拖慢讨论）
- 只用临时角色而不利用团队中已有的专业子代理

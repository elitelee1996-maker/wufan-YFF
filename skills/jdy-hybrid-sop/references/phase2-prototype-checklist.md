# 原型开发与验收检查清单（Phase 2）

> 本规范定义 JDY Hybrid 项目原型阶段的设计标准、质量要求和门禁条件。

---

## 1. Bridge 模式原型规范

Bridge 模式适用于需要与简道云深度集成的页面原型，使用 Fine Design 组件库模拟真实运行效果。

### 1.1 文件规范

| 要求 | 说明 |
|------|------|
| 文件格式 | **单文件 HTML**（所有 CSS/JS 内联，无外部依赖除 CDN） |
| 组件库 | 必须引入 Fine Design CDN：`https://cdn.fineres.com/design/latest/fine-design.min.css` + `.js` |
| Mock 数据 | 内嵌于 `<script>` 标签中，使用 `const MOCK_DATA = {...}` 格式 |
| 文件命名 | `proto_{模块}_{功能}.html`，如 `proto_purchase_order-list.html` |
| 字符编码 | UTF-8 |

### 1.2 Mock 数据规范

```javascript
const MOCK_DATA = {
  // 列表数据
  orderList: [
    { id: "ORD202401001", supplier: "XX供应商", amount: 15000, status: "pending", createTime: "2024-01-15" },
    // ...至少 5 条覆盖不同状态
  ],
  // 表单选项
  statusOptions: [
    { value: "draft", label: "草稿" },
    { value: "pending", label: "待审批" },
    { value: "approved", label: "已通过" },
    { value: "rejected", label: "已驳回" }
  ],
  // 当前用户上下文
  currentUser: { userId: "u001", name: "张三", role: "warehouse_admin" }
};
```

### 1.3 交互要求

| 交互类型 | 要求 |
|----------|------|
| 页面跳转 | 使用锚点或 JS 切换视图，**禁止打开新窗口** |
| 表单提交 | 点击后显示 loading → success/error toast，**不实际请求后端** |
| 列表筛选 | 基于 MOCK_DATA 做前端过滤，验证筛选逻辑正确性 |
| 弹窗/抽屉 | 使用 Fine Design Modal/Drawer 组件，含打开/关闭动画 |
| 状态流转 | 按钮根据 MOCK_DATA 中的状态动态显隐/禁用 |
| 响应式 | 至少适配 PC（1440px）和平板（768px）两个断点 |

## 2. Standard 模式原型规范

Standard 模式适用于纯自定义开发页面的原型，不强制使用 Fine Design。

### 2.1 基本要求

| 要求 | 说明 |
|------|------|
| 文件格式 | 单文件 HTML 或 Figma/Axure 导出链接 |
| 视觉风格 | 与客户品牌一致，标注色值、字号、间距 |
| 标注完整 | 所有交互元素需标注行为说明（hover/click/disabled） |
| 异常状态 | 必须包含空状态、加载态、错误态三种设计 |

### 2.2 交付物清单

- [ ] 原型文件（HTML 或设计工具链接）
- [ ] 交互说明文档（每个页面的操作→反馈映射）
- [ ] 视觉标注稿（色值、字号、间距、圆角）
- [ ] 响应式断点说明

## 3. 原型质量检查清单（8 项）

> 每个原型文件交付前必须逐项自检

| # | 检查项 | 标准 | 通过 |
|---|--------|------|------|
| 1 | **Mock 数据完整性** | 覆盖正常、边界、异常至少 3 类场景，列表 ≥ 5 条 | ☐ |
| 2 | **交互闭环** | 所有可点击元素均有反馈（toast/跳转/状态变化），无死链 | ☐ |
| 3 | **状态覆盖** | 包含空状态、加载中、成功、失败、无权限 5 种状态 | ☐ |
| 4 | **文案一致性** | 按钮/提示/标题用语统一，无错别字，术语与需求文档一致 | ☐ |
| 5 | **布局对齐** | 使用栅格系统，间距为 4px 倍数，无肉眼可见错位 | ☐ |
| 6 | **响应式验证** | PC + 平板两个断点下布局正常，无溢出/截断 | ☐ |
| 7 | **无障碍基础** | 表单有 label，按钮有语义化文本，颜色对比度 ≥ 4.5:1 | ☐ |
| 8 | **浏览器兼容** | Chrome 最新版 + Edge 最新版渲染一致 | ☐ |

## 5. 冻结原则说明

原型通过 G2 门禁后即进入**冻结状态**，遵循以下原则：

### 5.1 冻结含义
- 原型作为后续开发、测试、验收的**唯一 UI 基准**
- 任何偏离原型的实现均视为缺陷，除非走变更流程

### 5.2 变更流程
1. 提出方填写《原型变更申请单》，说明变更原因、影响范围
2. BA 评估对需求规格书的影响，UX 评估工作量
3. PM 审批（重大变更需客户签字）
4. UX 更新原型并重新通过质量检查清单
5. 更新版本号，通知开发和测试团队

### 5.3 允许的微调（无需变更流程）
- 文案修正（错别字、语序调整）
- 颜色微调（≤ 10% 色值差异）
- 间距微调（≤ 4px）
- Bug 修复导致的布局调整

### 5.4 违规处理
- 未经变更流程擅自修改原型 → 回滚 + 记录偏差
- 开发实现与冻结原型不一致且未报备 → 视为缺陷，限期修复

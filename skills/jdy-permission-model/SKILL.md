# 简道云权限体系与二开数据隔离

> 本技能定义简道云权限体系的完整模型，识别二开场景下的权限断裂点，并提供权限重建的标准方案。任何涉及数据隔离的评估都必须参照本技能。

## 一、权限体系全景

### 1.1 权限层级（纵向 6 层）

| 层级 | 名称 | 控制对象 | 配置入口 |
|------|------|---------|---------|
| L1 | **应用权限** | 谁能进入这个应用 | 应用设置 → 权限组 |
| L2 | **表单权限** | 谁能新建/查看/编辑/删除表单数据 | 表单设置 → 权限组 |
| L3 | **数据权限** | 谁能看到哪些数据行 | 表单设置 → 数据范围 |
| L4 | **字段权限** | 谁能看到/编辑哪些字段 | 表单设置 → 字段权限 |
| L5 | **流程权限** | 谁能提交/审批/回退/转交 | 流程设计器 → 节点负责人 |
| L6 | **仪表盘权限** | 谁能看到哪些报表/数据看板 | 仪表盘设置 → 权限组 |

### 1.2 权限主体（横向 3 种）

| 主体 | 标识 | 粒度 | 典型用法 |
|------|------|------|---------|
| **角色** | `role_no` | 功能分组 | "采购经理"可审批采购单 |
| **部门** | `dept_no` | 组织架构 | "华东区"只能看华东区数据 |
| **个人** | `username` | 精确指定 | "张三"是某流程的固定审批人 |

### 1.3 数据范围规则（核心机制）

| 数据范围 | 含义 | 底层过滤逻辑 |
|---------|------|-------------|
| **本人创建** | 只看自己创建的数据 | `creator.username = 当前用户` |
| **本人负责** | 只看自己负责的数据 | 表单中"成员字段"匹配当前用户 |
| **本部门** | 看本部门所有人创建的数据 | `creator.departments` 包含用户部门 |
| **本部门及下级** | 看本部门及所有子部门的数据 | 递归部门树匹配 |
| **全部数据** | 看所有数据 | 无过滤（管理员或特定角色） |
| **自定义规则** | 按字段值动态过滤 | 条件表达式（如"区域=华东"） |

### 1.4 权限组机制

简道云通过**权限组**将上述三层组合：

```
权限组 = 主体（角色/部门/个人） + 数据范围 + 操作权限（查看/新建/编辑/删除）
```

一个表单可以有多个权限组，用户命中任一权限组即获得对应权限。权限组之间是**并集**关系。

## 二、硬性约束（评估铁律）

| # | 约束 | 影响 |
|---|------|------|
| **P1** | **API Key 是上帝视角**：API 可读取所有人数据，绕过原生数据范围过滤 | 二开服务必须在应用层重建数据过滤，不能依赖简道云原生权限 |
| **P2** | **API 无法以用户身份操作**：所有 API 调用的身份是 API Key 持有者（通常是管理员） | 写操作需通过 `data_creator` 参数指定真实用户 |
| **P3** | **OAuth2.0 是唯一合规的身份透传方式** | iframe 嵌入场景必须走 OAuth，不能用 API Key 代替 |
| **P4** | **自定义详情页的数据源过滤基于当前登录用户** | 在原生界面内权限正确，但按钮跳转到外部页面时权限链条断裂 |
| **P5** | **流程操作的 username 必须与 task_id 对应** | 不能代替非当前节点审批人执行操作 |
| **P6** | **API 不能修改权限配置** | 权限组、数据范围、字段权限只能在可视化界面手动配置 |

## 三、各范式权限断裂点分析

### 3.1 断裂点矩阵

| 范式 | 权限状态 | 断裂点 | 风险等级 |
|------|---------|--------|---------|
| **A：标准表单** | ✅ 原生完整 | 无 | 🟢 无风险 |
| **B：流程驱动** | ✅ 原生完整 | 无 | 🟢 无风险 |
| **C：自定义详情页** | ⚠️ 基本完整 | 按钮跳转外部页面时身份丢失 | 🟡 低风险 |
| **D：集成应用** | ⚠️ 部分断裂 | 集成应用层需自行实现数据过滤 | 🟡 中风险 |
| **E：AI 增强** | ✅ 原生完整 | AI 功能在原生界面内运行 | 🟢 无风险 |
| **L3：智能助手 Pro** | ⚠️ 部分断裂 | HTTP 请求节点以系统身份执行外部调用 | 🟡 中风险 |
| **L4：完整后端** | ❌ 完全断裂 | 后端服务必须完整重建权限模型 | 🔴 高风险 |

### 3.2 断裂点详解

#### 断裂点 1：API 身份穿透（影响 L4）

```
原生界面：用户A登录 → 系统识别身份 → 按数据范围过滤 → 只看到A的数据 ✅
二开服务：API Key 调用 → 系统识别为管理员 → 看到所有数据 ❌
```

**后果**：用户A通过二开页面能看到本不属于他的数据。

#### 断裂点 2：iframe 身份丢失（影响 C/D）

```
无 OAuth：iframe 不知道当前用户 → 无法做权限过滤 ❌
有 OAuth：iframe 拿到用户身份 → 可在外部服务层做过滤 ⚠️（需自行实现）
```

#### 断裂点 3：data_creator 误用（影响 L4）

```
正确：jdy_create_data(data_creator="userA") → 数据创建者是 userA ✅
错误：jdy_create_data(不传 data_creator) → 数据创建者是 API Key 管理员 ❌
```

**后果**：数据归属错误，导致"本人创建"数据范围过滤失效。

#### 断裂点 4：智能助手 HTTP 请求（影响 L3）

```
智能助手触发 HTTP 请求 → 以系统身份调用外部服务
外部服务收到请求时不知道是哪个用户触发的 ⚠️
```

**缓解**：在 HTTP 请求的 body/header 中传递触发用户的 username。

## 四、权限重建方案（L4 二开场景）

### 4.1 三层权限重建架构

```
┌─────────────────────────────────────────────────────┐
│  第一层：身份识别（Identity）                          │
│  OAuth2.0 获取当前用户 username                       │
│  → 调用 jdy_get_corp_user 获取角色、部门信息           │
│  → 缓存 session：username + departments + roles       │
├─────────────────────────────────────────────────────┤
│  第二层：数据过滤（Data Filter）                       │
│  根据用户的数据范围，在 API 查询时注入 filter 条件      │
│  → "本人创建" → filter: creator.username = 当前用户   │
│  → "本部门" → filter: creator.dept contains 用户部门  │
│  → "自定义规则" → filter: 按规则表达式构建             │
├─────────────────────────────────────────────────────┤
│  第三层：操作鉴权（Operation Auth）                    │
│  写操作前检查用户是否有权限                             │
│  → 新建：检查"新建"权限                               │
│  → 编辑：检查是否是数据创建人/负责人                    │
│  → 删除：检查"删除"权限                               │
│  → 审批：检查 username 与 task_id 对应关系             │
└─────────────────────────────────────────────────────┘
```

### 4.2 身份层实现要点

| 要点 | 说明 |
|------|------|
| 必须使用 OAuth2.0 | 不能用 API Key 代替用户身份（P3 约束） |
| 登录后缓存用户信息 | `username` + `departments` + `roles`，避免每次请求都查 API |
| Session 过期策略 | 建议与简道云登录态对齐（通常 8 小时） |
| 多租户隔离 | 不同企业的 `role_no` 可能重复，需用 `app_id` 隔离 |

### 4.3 数据层实现要点

| 要点 | 说明 |
|------|------|
| 每次查询必须带 filter | 不能裸查全表（P1 约束） |
| filter 必须与原生数据范围一致 | 维护"用户-数据范围"映射表 |
| 分页查询的 filter 一致性 | 翻页时 filter 条件不能丢失 |
| 关联查询的权限传递 | 主表有权限不代表关联表也有权限 |

**filter 注入示例**：

```python
# 用户A的数据范围是"本人创建"
filter = {
    "rel": "and",
    "cond": [
        {"field": "_creator", "type": "text", "method": "eq", "value": ["userA"]}
    ]
}

# 用户B的数据范围是"本部门"（部门编号 12）
filter = {
    "rel": "and",
    "cond": [
        {"field": "_creator_dept", "type": "number", "method": "eq", "value": [12]}
    ]
}
```

### 4.4 操作层实现要点

| 操作 | 鉴权检查 | 实现方式 |
|------|---------|---------|
| **新建** | 用户是否有"新建"权限 | 检查权限组配置 |
| **编辑** | 用户是否是数据创建人/负责人 | 先 `jdy_get_data` 查 creator，再比对当前用户 |
| **删除** | 用户是否有"删除"权限 | 检查权限组配置（删除权限通常仅限管理员） |
| **审批** | 用户是否是当前节点审批人 | `jdy_get_flow_instance` 检查 operator 列表 |
| **回退** | 用户是否有回退权限 | `jdy_get_flow_instance` 检查当前节点配置 |

### 4.5 data_creator 使用规范

```python
# ✅ 正确：指定真实用户
jdy_create_data(
    app_id="xxx",
    entry_id="xxx",
    data={...},
    data_creator="userA"  # 数据归属 userA
)

# ❌ 错误：不传 data_creator
jdy_create_data(
    app_id="xxx",
    entry_id="xxx",
    data={...}
    # 数据归属 API Key 管理员，导致权限过滤失效
)
```

## 五、权限隔离检查清单

> 评估任何涉及数据隔离的项目时，按以下顺序逐项检查：

### Step 1：判断权限复杂度

- [ ] 是否有多角色（3 种以上）？→ 需要权限组设计
- [ ] 是否有部门级数据隔离？→ 需要数据范围规则
- [ ] 是否有字段级权限控制？→ 需要字段权限配置
- [ ] 是否有跨部门数据共享？→ 需要自定义数据范围规则

### Step 2：判断二开对权限的影响

- [ ] 是否通过 API 读写数据？→ 必须重建数据过滤层（P1）
- [ ] 是否有 iframe 嵌入？→ 必须走 OAuth2.0（P3）
- [ ] 是否有自定义详情页跳转外部页面？→ 需要身份透传
- [ ] 是否有智能助手调用外部服务？→ 需要在请求中传递用户身份

### Step 3：验证权限重建方案

- [ ] 身份层：是否使用 OAuth2.0 而非 API Key？
- [ ] 数据层：每次 API 查询是否注入了 filter？
- [ ] 操作层：写操作前是否检查了用户权限？
- [ ] data_creator：新建数据时是否指定了真实用户？
- [ ] 流程操作：username 是否与 task_id 对应？

| 触发条件 | 验证动作 |

## 七、常见反例

### 反例 1：API 裸查全表

```
❌ 错误：二开服务调用 jdy_list_data 不带 filter，返回所有数据后在前端过滤。
   → API 返回的数据已经绕过了原生权限，前端过滤不可靠（用户可绕过前端直接调 API）。
   → 处置：标记 P0 阻断，要求在后端注入 filter。
```

### 反例 2：iframe 不走 OAuth

```
❌ 错误：iframe 嵌入外部页面，通过 URL 参数传递用户身份（如 ?user=zhangsan）。
   → URL 参数可被篡改，不是合规的身份透传方式。
   → 处置：标记 P0 阻断，要求走 OAuth2.0（P3 约束）。
```

### 反例 3：data_creator 遗漏

```
❌ 错误：二开服务代替用户创建数据，未传 data_creator 参数。
   → 数据的 creator 变成 API Key 管理员，导致"本人创建"数据范围过滤失效。
   → 处置：标记 P1 告警，要求所有创建操作必须传 data_creator。
```

### 反例 4：权限重建成本遗漏

```
❌ 错误：L4 二开项目估算了 API 对接和前端开发人天，但未估算权限重建人天。
   → 权限重建（三层架构）通常需要 3-5 人天，遗漏会导致成本严重低估。
   → 处置：标记 P1 告警，要求 cost-margin-analyst 补充权限重建成本。
```

# 典型操作流程

> 用户问"我想做 X，大致怎么做?" — 这里给主线流程。

## 核心理念: 对话式优先，手动兜底

平台的产品理念是 **"用户和 Agent 对话解决一切"**。每个流程都按一个铁律组织:

- **Agent 能做的事** → 在对话里直接帮用户做
- **Agent 做不到的事 (用户必须手动)** → 给到精确的入口和步骤
- **做不好时** → 走对应内核技能或反馈通道

不要上来就甩一份"去设置点这点那"的清单 — 那是把 Agent 的活推给用户。

---

## 流程 1: 建一个主 Agent

**主 Agent** = 用户对话入口的 Agent，能在 Agent 选择器里切到它。跟"伙伴 / 子 Agent"是两回事 (流程 2)。

### 何时是主 Agent，何时是伙伴

只有满足以下三条全部，是**主 Agent**:

- 用户**直接对话**它 (不是被另一个 Agent 调起来)
- 它**直接服务用户** (不是给主 Agent 当帮手)
- 它**有明确角色**，不是"啥都能聊的通用助手"

如果用户更像是要"一个能帮主 Agent 干 XX 子任务的小专家" → 跳到流程 2 走伙伴。

### Agent 在对话里能做的事

1. **摸需求**: 按需用 ask user question 工具，问这些 (按需，不要一次问 5 个):
   - 服务对象是谁? (自己 / 团队成员 / 外部客户)
   - 主要在哪用? (网页端 / 飞书 / 钉钉 / 企业微信)
   - 主要工作内容?
   - 需要哪些连接能力? (要不要接外部服务)

   > **注意**: 用户问"能不能让外部系统通过 API 调用这个 Agent" → 老实说"目前 UI 没开放给用户配置外部调用 API Key，建议在小精灵反馈面板提一条让团队优先排"。

2. **设计蓝图**: 按需 (不一定都要) 列出
   - **灵魂** = 系统提示词 (agents.md 内容)
   - **工具选型** — 已有的复用，要新建的列出来
   - **连接凭证** — 调外部服务的提前想好
   - **技能** — 有方法论需求就建技能
   - **频道** — 用户要在 IM 里 @bot 才需要

3. **真实造资产**: 蓝图里要造的资产，调对应内置技能在对话里直接做:

   | 蓝图里要做的事 | 调用哪个内置技能 |
   |---|---|
   | 新建工具 / 配置连接凭证 / 写脚本读凭证 | `tool-connect` (工具与连接) |
   | 安装 MCP server | `mcp-installer` (MCP 服务安装) |
   | 沉淀方法论技能 | `skill-creator` (技能创建) |
   | 配伙伴 (子 Agent) | `subagent-creator` (伙伴创建) |

4. **交付**: 把所有材料备齐，让用户**最少几下点击**就能用起来:
   - 新建主 Agent 的入口路径
   - 要装备的资产清单
   - 系统提示词 (agents.md) **完整文本**
   - 3-5 句**快速测试用例**

### Agent 做不到，必须用户手动操作

主 Agent 实体本身的创建、装备、agents.md 粘贴、版本切换、发布**目前都是用户操作**。给到精确路径:

```
1. 设置 → 智能体资产 → "+ 新建 Agent"
   ├─ 名字 / 头像 / 简介
   └─ 默认 LLM 模型 (没接入自己的 → 用平台默认)

2. 在 Agent 配置页:
   ├─ 把 Agent 准备好的 agents.md 文本粘进去
   ├─ 装备资产 (技能 / 工具 / 伙伴 / MCP / 卡片)
   └─ 选发布策略 (流动版 / 受控版)

3. 在主对话区切换到这个 Agent，按 Agent 给的测试用例试聊
```

最低限度配置: 名字 + 默认模型 + agents.md。资产可以后面慢慢加。

### 做不好怎么办

试聊后用户反复说"不对" → 走 `references/task-troubleshoot.md`，定位是缺技能 / 缺记忆 / 模型不行 / 缺工具 / 风格不对，分别走不同补救路径。

## 流程 2: 建一个伙伴 (子 Agent)

**伙伴 / 子 Agent** = 主 Agent 通过 task 工具调的专家执行者。不直接面对用户。跟主 Agent 区别详见 `concepts.md`。

### Agent 在对话里能做的事

直接调用 `subagent-creator` 内置技能 — 这是平台为创建伙伴专门准备的技能，里面有完整的设计、写 agent.md、配工具集、能力边界设定的步骤。

### Agent 做不到，必须用户手动操作

把伙伴**装备到主 Agent**，目前是用户在 Agent 配置页操作:

```
1. 设置 → 智能体资产 → 伙伴 → "+ 新建伙伴" (如果不走对话方式)
2. Agent 配置页 → 伙伴标签 → 把建好的伙伴装备到主 Agent
3. 跟主 Agent 试一句对应任务，看它会不会自动调伙伴
```

### 做不好怎么办

主 Agent 不调伙伴的常见原因:
- 没装备到主 Agent
- 伙伴 description 写得不清楚 (主 Agent 不知道何时调它)
- 主 Agent 用的 LLM 模型不支持 function call

定位后回 `subagent-creator` 调整 agent.md 描述，或重新装备。

## 流程 3: 接入外部能力

用户说"我想让 Agent 接 X 服务" (X = 飞书 / GitHub / 自建 API / 内部 KMS / 企查查 / ...)。这是被问得**最多**的场景，也最容易选错路径。

### 第一步: 判断走哪条路

按以下 4 类做决策:

| 类型 | 适用场景 | 例子 | 在对话里调用的内置技能 |
|---|---|---|---|
| **a) 凭证独立 + 已有知识够用** | 一次性查询，Agent 在脚本里直接调 API 就能搞定 | "查杭州天气 (有 OpenWeather Key)" | `tool-connect` (只用凭证小节) |
| **b) 凭证 + 技能** | 不缺 API，缺**做事方法论**。Agent 不会做这类事，需要"教" | "Agent 写财报分析不专业" | `skill-creator` |
| **c) 凭证 + 工具** (单 / 多) | 把外部 API 沉淀成**长期可调用动作** | 单: "查公司工商信息工具" / 多: 一组飞书操作 | `tool-connect` 或 `mcp-installer` |
| **d) 凭证 + 工具 + 技能** | 工具能跑通但 Agent 用得不好，要**加方法论指挥工具** | "工具能查工商信息，但用户要全套尽调按规范输出" | 三件套联动 |

### a) 凭证独立 + 已有知识够用

用户的需求 Agent 当场写几行 Python 就能解决，不需要长期沉淀成工具。凭证存好，Agent 自己用 `execute_python` / `bash` + `httpx` / `curl` 调一下 API 即可。

**例子**:

> 用户: "我想让 Agent 帮我查实时天气，我有 OpenWeather 的 API Key"

→ 不用建工具。让用户在**连接凭证**里建一个 OpenWeather (字段 api_key, 团队级)。然后 Agent 后续问到天气时，在 `execute_python` 里读环境变量调 API 就行。

**为什么不建工具**: 一次性查询沉淀工具反而更重 (建工具要写 tool.json + 注册 + 装备 + 维护)。凭证存住，Agent 即刻可用。

**Agent 在对话里要做的**: 调用 `tool-connect` 内置技能里**"读取凭据的三种上下文"**那张表 — 在 `execute_python` / `bash` 里怎么读凭证那段，是这条路径的核心。

### b) 凭证 + 技能

这类需求**不在"调外部 API 不够"**，而是 Agent 缺一套**做这类事的方法论 / 工作流**。凭证可能用得到 (读内部数据源)，但核心补强是**技能**。

**例子**:

> 用户: "Agent 写财报分析每次都不专业 — 该看哪些指标 / 怎么交叉验证 / 怎么排版输出，它都不太懂"

→ 这是 Agent **缺方法论**。沉淀一个"财报分析"技能: 写清楚分析框架、关键指标、交叉验证方法、输出模板。Agent 触发关键词时 read 它就有专业的做事路径。

**Agent 在对话里要做的**: 调用 `skill-creator` 内置技能。把方法论沉淀成 SKILL.md + references。

**避免**: 不要把方法论零碎塞进 agents.md 的 system prompt。那会污染 system prompt 且不可复用。

### c) 凭证 + 工具 (单 / 多)

把外部 API 沉淀成**长期可调用的动作**。这是最常见的"接外部能力"形态。

**单 vs 多**:

| 场景 | 怎么做 |
|---|---|
| **只需要 1 个动作** | 自建单工具 |
| **一组相关动作 + 业界有现成 MCP** | 装 MCP (省事，一次拿一组) |
| **一组相关动作 + 没现成 MCP** | 自建工具包 (pack)，多个 tool.json 集合 |

**例子 — 单工具**:

> 用户: "Agent 经常要帮我查公司工商信息，每次都现写代码挺烦"

→ 沉淀 `qcc_company_query` 工具。凭证: 企查查 API Key (团队级)。工具: 一段 Python，入参公司名，出参 JSON。

→ 在对话里调用 `tool-connect` 内置技能完成。

**例子 — 多工具用 MCP**:

> 用户: "Agent 经常要操作飞书 — 发消息 / 建文档 / 查日历，每个都建工具太麻烦"

→ 装现成的飞书 MCP，一次拿到 20+ 个飞书工具。在对话里调用 `mcp-installer` 内置技能完成。

### d) 凭证 + 工具 + 技能

已经做了 c (有工具能调 API)，但**用户反馈 Agent 还是用得不好** — 不知道何时调哪个工具 / 不知道怎么组合调用 / 输出格式不一致。

**例子**:

> 用户: "工具我建了，凭证也配了，Agent 能查工商信息了。但每次它只查一项，我想要**全套尽调** (工商 + 股权 + 司法 + 经营异常 + 知识产权)，且按规范格式输出"

→ 单纯加工具解决不了，是**方法论不够**。在 c 基础上加一个技能，教 Agent 怎么组合调用这些工具 + 怎么排版输出。

**怎么做**:
- 已有工具与凭证 → `tool-connect` 已搞定
- 加方法论 → 调用 `skill-creator` 沉淀"公司尽调"技能，写清触发关键词、调哪些工具、什么顺序、怎么交叉验证、输出格式模板

**判断信号 — 何时该加技能**:
- 用户**反复纠正** Agent 怎么用工具
- 同类任务 Agent 表现**不稳定**
- Agent **不知道该不该调**这些工具，规则塞不进 system prompt

满足任一信号就加技能。

> 技能不是"工具的替代品"，是**工具的指挥官**。工具是动作，技能是"什么时候做哪个动作 + 做完怎么处理"。

### Agent 做不到，必须用户手动操作

- **新建连接凭证**: 设置 → 连接凭证 → "+ 新建连接凭证"
- **装备工具 / MCP / 技能到 Agent**: Agent 配置页 → 资产装备
- 用户撤销凭证授权 / 切换字段所有权 / 删除整个连接

### 做不好怎么办

凭证配齐了但调用还报错，按以下顺序排查 (写在 `references/connections.md` 末尾):
1. 工具是否真的装备到当前 Agent (enabled 状态?)
2. 凭证字段名跟工具代码里读的 key 是否一致
3. 字段是 team 还是 user — user 级要看**当前用户**有没有填
4. API Key 本身是否过期 / 被吊销

## 流程 4: 邀请成员到团队空间

**只有 owner / admin 能邀**。这是纯设置页操作，没有对话式版本。

```
1. 设置 → 成员管理 (没看到这一项 → 你不是 owner / admin，找空间 owner)
2. "邀请成员"按钮
3. 输入邮箱 / 用户名 / 帆软通行证 ID，选角色 (member / admin / viewer)
4. 平台发邀请: 已注册 → 直接加入 / 未注册 → 邮件邀请注册
```

**容量**: 默认上限 30 成员，付费订阅可大幅扩容。配额满会报"团队成员数量已达上限"，让用户在订阅页升级。

**注意**:
- admin 不能邀新的 admin (只能邀 member 或 viewer)
- 一个用户已经在团队中再次邀请会报错

## 流程 5: 切换 Agent 的版本 (流动版 ↔ 受控版)

**何时换流动版 → 受控版**: Agent 准备给团队 / 外部用，不想小改动直接生效，要"调试稿 vs 正式稿"区分。

**何时换受控版 → 流动版**: Agent 是个人玩 / 内部工具，受控版每次发版手动太累。

```
1. Agent 配置页 → "发布策略"按钮
2. 切到流动版 / 受控版
3. 切到受控版后:
   ├─ 当前所有改动留在草稿态
   ├─ 普通成员立刻看到"上次发布的稳定版"，没发过则看不到
   └─ 第一次发布: Agent 配置页 → "发布"按钮
4. 切回流动版后:
   └─ 草稿改动立刻生效，之后任何改动直接发，没有"草稿"概念
```

**用户报"我做的修改 Agent 没生效"** — 大概率是受控版下没发布。让用户找"发布"按钮。

## 流程 6: 接入自己的大模型 (BYOK)

**何时**: 用户想用自己买的 API Key (省钱 / 用特定模型 / 隐私要求)。

```
1. 设置 → 大模型接入 → "+ 添加模型"
2. 选 provider (OpenAI / Anthropic / DeepSeek / 智谱 / 月之暗面 / 豆包 / Ollama / 自定义)
3. 填:
   ├─ 显示名 (你给这个模型起的名字，例: "我的 GPT-4")
   ├─ 模型 ID (如 gpt-4-turbo)
   ├─ API Key
   ├─ Base URL (用代理 / 自建端点时填；官方默认就别填)
   └─ 共享方式: 团队共享 / 个人接入
4. 保存。模型立刻进可选列表
5. 在对话输入框上方的模型选择器切到它，或在 Agent 配置页设为默认模型
```

**用户问"接的模型为啥用不了"** — 排查:
- API Key 错 / 过期
- provider 选错 (DeepSeek 不能选 OpenAI provider)
- Base URL 必填没填 (自建 endpoint 一定要填)
- 共享方式: user 级要自己填，team 级要 owner 配过

## 流程 7: 分享文件 / 分享会话

> 重要: 平台的分享**都是永久有效的**，**没有过期时间选项**。不要凭"行业经验"瞎说"30 天 / 7 天有效" — 那是错的。

### 分享文件

```
1. 在工作室面板或文件预览页右上的"分享"按钮
2. 自动生成永久公开链接
3. 拷贝链接发给别人，访客无需登录就能看
```

### 分享会话

```
1. 聊天页面右上角的"分享"按钮
2. 配置:
   ├─ 标题 / 介绍 (访客打开看到的)
   └─ "是否允许查看上传文件":
       ├─ 默认勾选: 用户上传给 Agent 的文件能看
       └─ 取消: 只看到对话，看不到原始上传文件
3. 自动生成永久公开链接
4. 拷贝链接发给别人。访客看到回放 + 工作室面板里 Agent 产出的文件
```

**会话分享**支持回放 (消息一条条回放) + 工作室产出文件预览，适合给同事看你跟 Agent 怎么解决问题的全过程。

## 一行心法

每个流程**先想"能不能让 Agent 在对话里直接帮用户做完"**:
- 能 → 走对话路径，调用对应内置技能 (`tool-connect` / `skill-creator` / `mcp-installer` / `subagent-creator`) 主导执行
- 不能 (用户必须手动) → 给精确入口和步骤，别只说"在设置里"
- 试过做不好 → 走 `task-troubleshoot.md` 找改进方向，或推用户提反馈

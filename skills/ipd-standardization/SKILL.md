# IPD 需求标准化指南

## 核心层级定义
- **L1 (流程域)**: 代表核心业务领域（通常 3-8 个）。示例：供应链采购管理、供应商全生命周期。
- **L2 (流程组)**: 代表流程分组（每个 L1 下 2-5 个）。示例：供应商引入管理、供应商退出管理。
- **L3 (流程)**: 代表具体业务流程（每个 L2 下 2-6 个）。示例：供应商注册、资质审查、现场评审。
- **L4 (节点)**: 代表最小业务单元（每个 L3 下 1-4 个）。示例：基础信息填报、资质到期预警。

## 字段类型参考（🆕 V10.7.0-V10.20.0 新增）

拆解 L4 节点时，需考虑以下新增字段类型：

| 字段类型 | 上线版本 | 适用场景 |
|---------|---------|---------|
| **富文本字段** | V10.7.0 | 需要格式化文本、图文混排的场景（支持搜索、筛选，数量上限提升） |
| **计算字段** | V10.7.0 | 需要实时计算的数值场景（公式自动计算，无需智能助手） |
| **关联子表字段** | V10.0.0 | 需要在主表中直接展示关联子表数据的场景（替代部分子表单需求） |

> 完整字段类型列表见 `jdy-custom-dev-capabilities` 技能的字段类型映射表。

## 输出格式规范（🆕 v2.0 重构）

### 唯一输出格式：4-Sheet Excel（`需求报价清单_[项目名].xlsx`）

> **模板参考**：用户上传的「双瑞海洋项目运营管理平台_需求报价清单_最新规则.xlsx」（4-Sheet 结构，含业务需求清单/项目评估报价/项目风险清单/项目运维服务）。子 Agent 执行前必须 `read_document` 读取该文件作为格式参照。

#### Sheet 1：业务需求清单

| 列名 | 说明 | 数据来源 |
|------|------|---------|
| **编号** | 从 1 开始自增 | — |
| **功能模块** | `L1 流程域 → L2 流程组`，如「项目立项与主数据」 | L1+L2 合并 |
| **功能名称** | L3 流程名，一行一个 L3 | L3 |
| **需求说明** | L4 节点展开，格式：`【节点名】业务描述 / 输入条件 / 处理逻辑 / 输出结果 / 关联角色` | L4 |
| **标准配置人天** | 零代码配置人天（按 `cost-estimation-model` 基准） | 阶段1 预估 |
| **接口/开发人天** | 涉及外部系统联调、二开的人天 | 阶段1 预估 |
| **合计人天** | `= 标准配置人天 + 接口/开发人天` | 公式 |
| **报价理由（内部）** | 简要说明人天依据，如「标准流程表单，2天」 | 阶段1 填写 |

末尾保留**合计行**：`功能需求合计（建设人天）`。

#### Sheet 2：项目评估报价

| 列名 | 说明 |
|------|------|
| **编号** | 阶段编号 |
| **项目阶段** | 固定 7 阶段：项目启动 / 项目规划 / 系统实现 / 系统培训 / 系统上线 / 系统验收 / 项目管理 |
| **阶段任务** | 具体任务描述 |
| **交付物** | 该阶段产出物 |
| **人天工作量** | 从 Sheet1 汇总 + 阶段活动经验值 |
| **服务方式** | 远程 / 现场 / 代码开发 |
| **费用单价** | 默认 2500 元/天 |
| **费用** | `= 人天 × 单价` |

末尾保留**合计行**和**项目运维（每年）**行。

#### Sheet 3：项目风险清单

| 列名 | 说明 |
|------|------|
| **序号** | 从 1 开始 |
| **风险分类** | 集成 / 范围 / 协同 / 附件 / 报表 / 计划 / 权限 / 数据 / 交付 / 联调 |
| **风险项** | 风险名称 |
| **风险说明** | 具体描述 |
| **风险等级** | 高 / 中 / 低 |
| **建议应对措施** | 可操作的应对方案 |
| **责任方** | 客户侧 / 实施方 |
| **确认节点** | 方案确认前 / Demo 阶段 / 蓝图设计 / 上线准备前 |

⚠️ **阶段1 留空**：Sheet 3 的风险清单由阶段2（经验检索）反哺填写。阶段1 的 `requirement-analyst` 只产出 Sheet 1（业务需求清单），Sheet 3/4 由主 Agent 在阶段2 检索完成后合并填入。后续阶段可追加新风险。

#### Sheet 4：项目运维服务

| 列名 | 说明 |
|------|------|
| **序号** | 从 1 开始 |
| **服务类别** | 表单维护 / 流程维护 / 权限维护 / 问题处理 / 稳定性保障 / 现场巡检 |
| **服务内容** | 具体服务范围描述 |
| **服务方式** | 远程服务 / 现场服务 |

固定模板，按项目类型微调。

## 行为准则（🔄 已更新）
1. **只输出 L1-L4**: L5 属于项目实施阶段细化内容。
2. **描述准确性**: L4 节点说明必须具体可操作，避免空泛。
3. **比对标注**: 明确标注哪些是标准品能力，哪些是客户自定义扩展。
4. **AI 能力匹配**（🆕）: 拆解 L4 节点时，检查是否可通过 AI 功能（推荐字段/填充/执行节点/生成公式）零代码实现。标注"AI 可实现"的节点不计入二开范围。
5. **插件优先检查**（🆕）: 涉及 ERP/签章/发票/硬件/电商/打印等集成场景时，先检查插件市场是否有现成方案，标注"插件可实现"的节点不计入二开范围。
6. **自定义详情页优先**（🆕）: 涉及定制化数据展示需求时，优先评估自定义详情页（范式 C）是否满足，标注"自定义详情页可实现"的节点不计入二开范围。
7. **权限需求标注**（🆕）: L4 节点涉及数据读写时，必须标注权限需求：
   - 标注"数据范围"：本人创建/本人负责/本部门/本部门及下级/全部数据/自定义规则
   - 标注"操作权限"：查看/新建/编辑/删除/审批/回退
   - 标注"二开权限风险"：若该节点需要 L4 二开，必须标注"需权限重建"（API Key 是上帝视角，绕过原生权限）
   - 标注"身份透传"：若涉及 iframe 嵌入，必须标注"需 OAuth2.0"
8. **人天预估**（🆕）: 拆解 L3 节点时，必须同步填入预估人天：
   - 加载 `cost-estimation-model` 技能的人天基准（普通表单 0.5-1天、流程 1-2天、仪表盘 2-4天、API 接口 3-5天、前端自定义页 5-10天、权限重建 3-5天）
   - 判定是否需要接口/二开（有无外部系统集成），填入「接口/开发人天」列
   - 标注「⚠️ 预估，阶段4 精确测算」
9. **阶段分工**（🆕）: 阶段1 只产出 Sheet 1（业务需求清单），Sheet 3（风险清单）和 Sheet 2/4 由主 Agent 在阶段2 经验检索完成后合并填入。原因：风险应以历史案例和产品能力边界为主要来源，而非凭空撰写。

## 关联技能（🔄 已更新）
- **jdy-custom-dev-capabilities**: IPD L3/L4 节点拆解后，必须用该技能的"评估决策检查表"逐项检查是否超出平台能力边界。硬性约束 H1-H11 是判定节点可行性的铁律。**新增插件生态检查（6.1）和自定义详情页检查（3.1）**。
- **jdy-architecture-patterns**: IPD 节点确定后，用架构范式判定法决定哪些层用简道云原生、哪些需要二开。**新增范式 C/D/E 和 AI 能力匹配步骤**。
- **cost-estimation-model**: 节点可行性确认后，用人天基准估算成本。**新增自定义详情页（2-4人天/页）、AI功能（1-2人天/项）、插件配置（0.5-1人天/个）、权限重建（3-5人天/项目）基准**。
- **jdy-permission-model**（🆕）: L4 节点涉及二开数据读写时，必须用该技能的"断裂点矩阵"评估权限风险，用"三层权限重建架构"设计方案。权限重建是最常见的成本低估原因。

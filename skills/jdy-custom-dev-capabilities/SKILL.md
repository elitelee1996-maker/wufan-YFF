# 简道云二开定制能力边界与硬性约束

> 本技能定义简道云平台的**能力天花板**。任何需求评估、架构设计、成本估算都不得违反本文中的硬性约束。

## 一、平台定位

简道云是**零代码协作流程数据收集平台**，前端可定制部分有限。层级结构：`应用 → 表单 | 仪表盘 | 流程表单`。

---

## 二、硬性约束清单（评估铁律）

> 以下约束为**不可突破的平台限制**。评估时遇到任何一条被违反，必须标记为 🔴 P0 阻断。

### 2.1 结构与数据约束

| # | 约束 | 影响 |
|---|------|------|
| H1 | **API 不能修改表单结构**（不可增删改字段） | 表单结构变更只能在可视化界面手动完成，不能通过自动化脚本批量建表 |
| H2 | **子表单最多 200 行** | 超过 200 行的明细需求必须拆表或用外部系统 |
| H3 | **部分字段 API 不可写入**：分割线、手写签名、选择数据/查询、流水号 | 这些字段只能通过界面操作，不能通过 API 导入 |
| H4 | **附件/图片 URL 有效期 15 天** | 不能将简道云文件 URL 作为永久链接使用 |
| H5 | **文件上传 token 有效期 1 小时** | 上传流程必须在 1 小时内完成 |

### 2.2 性能约束

| # | 约束 | 影响 |
|---|------|------|
| H6 | **API 全局频率 50 次/秒** | 大批量数据操作需要排队和限流 |
| H7 | **单接口频率 10-30 次/秒**（数据接口 30，表单查询 30，批量操作 20） | 批量写入 10000 条数据至少需要 ~17 秒（100条/次 × 20次/秒） |
| H8 | **后端插件执行限时 1 分钟** | 超时任务必须拆分或放到外部服务 |
| H9 | **后端插件沙盒环境**：无磁盘 IO、有内存限制 | 不能做文件处理、大数据计算等重操作 |

### 2.3 身份与权限约束

| # | 约束 | 影响 |
|---|------|------|
| H10 | **API 可读取所有人数据，但无法以该人身份操作** | 需要身份隔离的场景必须在外部服务层实现权限控制 |
| H11 | **OAuth2.0 可将身份带到外部站点和 iframe** | 这是唯一合规的身份透传方式 |

## 三、可定制能力全景

### 3.1 前端可定制部分（🔄 已更新）

| 能力 | 说明 | 典型场景 |
|------|------|---------|
| **自定义详情页**（🆕 V10.10.0+） | 简道云原生定制展示页面，支持图表/列表/简报卡片/按钮/打印模板/组件显隐规则/移动端/多数据源 | 高度定制化数据展示，**可替代部分 iframe 场景**（2-4 人天/页） |
| **OAuth2.0 身份鉴权** | 将简道云身份带到外部站点或 iframe | 单点登录到定制页面 |
| **仪表盘嵌入 iframe** | 嵌入固定链接 | 挂载自研看板、数据大屏 |
| **数据详情页嵌入 iframe** | 根据数据字段拼接链接（仅支持 `?no=xxx` 一种传参方式） | 查看外部系统关联数据 |
| **弹窗交互** | 仅单条记录编辑模式可用，可传字段、返回字段 | 搜索外部数据库、调用第三方服务 |

> ⚠️ **决策建议**：优先评估自定义详情页（范式 C）是否满足需求，再考虑 iframe 方案。自定义详情页是简道云原生能力，配置成本低、维护简单、支持移动端。

### 3.2 后端可定制部分

| 能力 | 说明 | 限制 |
|------|------|------|
| **后端插件（Python/JS）** | 云函数，可由表单事件/智能助手/前端触发 | 1 分钟限时、沙盒、无磁盘 IO |
| **智能助手** | 记录提交/更新/定时触发的自动化流 | 可串联后端插件节点 |
| **Webhook** | 数据增删改恢复时推送 HTTP 通知 | 推送到外部定制服务 |

### 3.3 API 能力矩阵（🔄 已更新）

| API 类别 | 读 | 写 | 关键限制 |
|---------|:---:|:---:|---------|
| 应用/表单结构 | ✅ | ❌ | 只能查，不能改结构 |
| 数据 CRUD | ✅ | ✅ | 批量最多 100 条/次，频率 20-30 次/秒 |
| 文件上传下载 | ✅ | ✅ | 两步式上传，token 1h，URL 15 天 |
| 流程操作 | ✅ | ✅ | 提交/回退/转交/加签/撤回/否决 |
| 通讯录 | ✅ | ✅ | 成员/部门/角色的 CRUD |
| **资源用量**（🆕 V11.2.0） | ✅ | ❌ | 查询应用/表单的资源使用情况 |
| **抄送列表**（🆕 V10.5.0） | ✅ | ❌ | 查询流程抄送记录 |
| **待办否决**（🆕 V9.15.0） | ❌ | ✅ | 否决待办任务（终止流程） |

> 详细 API 规格见 `references/api-reference.md`
> 字段类型映射见 `references/field-type-mapping.md`

## 四、七大触发流转模式

评估需求时，先判断属于哪种模式，再评估可行性：

| # | 模式 | 流转路径 | 适用场景 |
|---|------|---------|---------|
| T1 | 智能助手链 | 提交表单 → 智能助手 → 后端插件 → 外部服务 → 回写参数 | 需要自动处理和回写的业务 |
| T2 | Webhook 推送 | 提交表单 → Webhook → 外部服务 | 数据变更通知、实时同步 |
| T3 | 弹窗交互 | 编辑记录 → 点击按钮 → 弹窗 iframe → 返回字段 | 搜索外部数据、调用第三方服务 |
| T4 | 仪表盘嵌入 | 打开应用 → 仪表盘 → 内嵌页面 | 自研看板、数据大屏 |
| T5 | 外部回写 | 外部触发 → 定制服务 → API 写入简道云 → 消息提醒 → 自动提交流程 | 外部系统驱动简道云 |
| T6 | 链接导航 | 记录新建 → 设置唯一链接 → 点击链接 → 导航到定制服务 | 查看关联外部数据 |
| T7 | 数据详情页 | 打开记录 → 数据详情页 iframe → 嵌入外部页面 | 单条记录的外部数据展示 |
| **T8** | **双模式协同**（🆕） | 数据汇聚页(二开) → 点击数据链接 → 简道云原生详情页(编辑) → 智能助手 → Webhook → Bridge重算 → 写回简道云 → 汇聚页刷新 | 需要全局聚合视图 + 单条记录深度交互的场景（甘特图+任务详情、资源日历+资源详情） |

> **T8 与其他模式的区别**：T1-T7 都是单向流转（简道云→外部 或 外部→简道云）。T8 是**双向闭环**——二开页面和简道云原生页面之间来回跳转，数据通过 Bridge 层保持同步。

## 五、四大开发模式（决策树）

### 模式 A：数据同步

**简道云 → 第三方**：Webhook 推送 或 API 定时轮询
**第三方 → 简道云**：定制服务抓取 → API 写入

适用：企业内部系统（打卡/任务/数据库）、第三方平台（WMS/ERP/签章/AI/机器人）、短信邮件

### 模式 B：能力增强

通过插件增强表单能力：编辑时触发前端/后端函数，实时查询外部数据并回填。

适用：工商信息查询、地址解析、实时计算等

### 模式 C：专有页面

结合外部页面增强表现力：
- 仪表盘嵌入：独立管理页面（OAuth2.0 单点登录）
- 数据详情页嵌入：单条记录交互
- 弹窗搜索：提交时搜索外部数据库回填子表单

适用：自定义打印、共享文档编辑、园区看板、复杂报表

### 模式 C 补充：简道云表单的双重角色（🆕 2026-06-14）

> **关键认知更新**：简道云表单**不只是数据库**，它同时也是前端交互闭环的一部分。

| 角色 | 说明 | 典型操作 |
|------|------|---------|
| **数据底座**（传统认知） | 表单存储业务数据，API 提供 CRUD | 二开页面通过 API 读写数据 |
| **交互前端**（🆕 新增认知） | 表单详情页本身就是交互界面，用户在这里编辑字段、触发流程、操作按钮 | 用户在简道云原生详情页完成编辑，智能助手自动触发后续处理 |

**设计影响**：
- 二开页面（模式 A）**不应自行实现** CRUD 表单编辑——应链接到简道云原生详情页（模式 B）
- 简道云详情页的数据变更**必须**通过智能助手/Webhook 通知 Bridge 层
- Bridge 层重新计算后**写回**简道云，确保二开页面刷新后数据一致

**判定标准**：如果一个操作只是"编辑单条记录的字段"，不需要二开——直接用简道云原生详情页。

### 模式 D：物理设备交互

边缘服务（连接设备）+ 云端服务（接收信息 + 下发指令 + 与简道云交互）

适用：地磅读取、血压仪测量、打印机控制、串口设备

## 六、集成能力（🔄 已更新）

| 能力 | 说明 |
|------|------|
| **单点登录** | 从外部平台（钉钉/企微/飞书）登录到简道云（OAuth1） |
| **人员+部门同步** | 从钉钉/企微/飞书读取通讯录同步到简道云 |
| **消息推送** | 待办/转交/抄送消息推送到钉钉应用消息或外部服务 |
| **待办同步** | 同步简道云待办到其他待办平台 |

> 注意：飞书部门 ID 为字符串，与简道云 dept_no（数字）不同，需映射处理。

### 6.1 插件生态（🆕 V10.0.0-V11.2.0 爆发式增长）

简道云插件市场已覆盖 **30+ 第三方平台**，评估时需优先检查是否有现成插件可用，避免重复二开：

| 类别 | 代表插件 | 替代场景 |
|------|---------|---------|
| **ERP 集成** | 金蝶云星空/星辰/旗舰版、用友 YonSuite/BIP/好会计、畅捷通 T+Cloud | 原本需要二开的 ERP 数据同步 |
| **AI 能力** | DeepSeek、豆包 AI、百度 AI 大脑、通义千问 | 原本需要二开的智能分类/填充/分析 |
| **电子签章** | e 签宝（次数版/年费版）、上上签、法大大 | 原本需要二开的合同签署流程 |
| **发票财税** | 数电发票、电子税务局进项/销项、发票识别/验真 | 原本需要二开的发票管理 |
| **硬件集成** | 海康威视考勤机、温湿度数据同步、电表管理平台 | 原本需要模式 D（物理设备）二开 |
| **电商** | 1688 订单售后、聚水潭、抖音来客 | 原本需要二开的电商数据同步 |
| **打印** | 快麦云打印（多次优化）、易云云打印 | 原本需要二开的自定义打印 |

> ⚠️ **评估铁律**：在判定需要二开之前，**必须**检查插件市场是否有现成方案。插件方案的成本（0.5-1 人天配置）远低于二开方案（3-15 人天）。

## 七、评估决策检查表

> 评估任何需求时，按以下顺序逐项检查：

### Step 1：能否零代码实现？（🔄 已更新）
- [ ] 标准表单 + 流程 + 仪表盘能否覆盖？→ 不需要二开
- [ ] **自定义详情页**能否满足展示需求？→ 不需要二开（范式 C，2-4 人天/页）
- [ ] **智能助手 Pro**（含 HTTP 请求/Agent/循环容器/AI 大模型节点）能否实现自动化？→ 不需要二开
- [ ] **AI 功能**（推荐字段/填充/执行节点/生成公式）能否实现智能处理？→ 不需要二开
- [ ] **插件市场**是否有现成方案（ERP/签章/发票/硬件/电商/打印）？→ 不需要二开（0.5-1 人天配置）
- [ ] **集成应用**（V11.0.0+）能否满足系统集成需求？→ 不需要二开
- [ ] **双模式协同**（🆕）：该页面是否只是"单条记录编辑"？→ 不需要二开，用简道云原生详情页 + 数据链接即可

### Step 2：需要哪种二开模式？
- [ ] 只是数据进出？→ 模式 A（数据同步）
- [ ] 需要增强表单交互？→ 模式 B（能力增强）
- [ ] 自定义详情页无法满足，需要更复杂的页面？→ 模式 C（专有页面，iframe 嵌入）
- [ ] 需要连接物理设备且无现成插件？→ 模式 D（物理设备）

### Step 3：硬性约束检查
- [ ] 是否需要通过 API 修改表单结构？→ ❌ 不可行（H1）
- [ ] 子表单是否超过 200 行？→ ❌ 需拆表（H2）
- [ ] 后端插件是否超过 1 分钟？→ ❌ 需外部服务（H8）
- [ ] 是否需要大批量数据实时同步？→ ⚠️ 受频率限制（H6/H7）
- [ ] 附件 URL 是否需要永久有效？→ ❌ 只有 15 天（H4）

### Step 4：成本影响评估
- 参考 `cost-estimation-model` 技能中的人天基准
- 数据同步类：3-5 人天/接口
- 前端自定义页面：5-10 人天/页
- 后端插件：2-4 人天/个
- 物理设备交互：8-15 人天/设备类型

## 八、与其他技能的协作关系

```
需求输入
  │
  ├─→ ipd-standardization：拆解为 L1-L4 节点
  │     │
  │     └─→ 本技能：逐节点检查是否超出平台能力
  │           │
  │           ├─→ jdy-architecture-patterns：判定架构范式（L0-L4 分层）
  │           │     └─→ 本技能的"硬性约束"验证架构方案可行性
  │           │
  │           └─→ cost-estimation-model：二开人天估算
  │                 └─→ 本技能的"开发模式"决定成本基线
  │
  └─→ 最终评估报告
```

### 关键交叉验证点（🔄 已更新）

| 触发条件 | 验证动作 |
|---------|---------|
| architect 判定"L3 层需二开" | 用本技能 Step 3 检查硬性约束 |
| cost-margin 估算了"前端自定义页面" | 用本技能 3.1 确认前端可定制范围是否覆盖，**优先评估自定义详情页替代方案** |
| IPD 中出现"批量导入 10000+ 条" | 用 H6/H7 计算实际耗时，标注 P1 |
| IPD 中出现"实时物理设备数据" | 用模式 D 评估边缘+云端双层架构，**先检查插件市场是否有现成方案** |
| 需求涉及"修改表单字段配置" | 用 H1 判定不可行，标注 P0 |
| **需求涉及"定制化数据展示"**（🆕） | **优先评估自定义详情页（范式 C），再考虑 iframe 方案** |
| **需求涉及"智能分类/填充/分析"**（🆕） | **优先评估 AI 功能（推荐字段/填充/执行节点），再考虑二开** |
| **需求涉及"ERP/签章/发票/电商集成"**（🆕） | **先检查插件市场是否有现成方案，再考虑二开** |
| **需求涉及"自动化流程"**（🆕） | **优先评估智能助手 Pro（HTTP 请求/Agent/循环容器/AI 大模型节点）的覆盖度** |

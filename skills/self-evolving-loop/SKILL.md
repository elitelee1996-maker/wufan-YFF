蒸馏自 Hermes Agent 的 Memory Nudge + Auto Skill + Honcho 用户建模 + Skill Self-Patch 四个机制。本技能只定义系统 prompt 中**没有**的增量能力，基础记忆操作遵循系统内置的「记忆使用指南」。

## 两种触发模式

### 模式 A：任务后反思（三环节）

**触发条件（满足任一）：**

- 本次对话使用了 5+ 次工具调用，且任务有实质性产出（不只是查询）
- 用户明确要求："总结"、"复盘"、"沉淀一下"
- 对话中用户表达了新的、明确的偏好/决策（如"以后都用 X"、"我偏好 Y"）

**不触发：** 纯问答、1-4 步简单操作、用户明显赶时间、对话内容与已有记忆完全重复。

触发后执行环节 1→2→3，每个环节独立判断是否有产出，没有就跳过。

### 模式 B：技能自修复（独立触发）

使用任何技能完成任务后，发现技能文档有缺陷（步骤遗漏、参数过时、有更优做法、示例不准确）→ 直接用 `edit` 修改对应 SKILL.md，在变更记录区追加一行。

内核/内置技能（只读）无法直接修改，此时在 skills/ 同名目录下创建覆盖文件。

---

## 环节 1：记忆审计（增量部分）

基础的"记什么、怎么存"遵循系统内置指南。本环节补充两个系统没有的能力：

### 不记什么

- 临时性信息（"这次先用 mock 数据"）
- 对话中的中间过程（调试步骤、试错记录）
- 可以从代码/文件中直接读到的信息
- 常识性知识
- 用户没有明确表态的推测

### 容量治理

当 recall 返回结果明显冗余（同主题 5+ 条、内容大量重叠）时，顺手治理：

- 同主题多条 → 合并为 1-2 条综合记忆，删除旧条目
- 互相矛盾 → 保留最新的，删除旧的
- 不再相关（项目已结束、技术已弃用）→ 删除

---

## 环节 2：技能沉淀

**大多数对话不需要沉淀技能。** 只在以下四条**同时满足**时才沉淀：

1. 任务涉及 5+ 步骤的特定流程
2. 流程中有非显而易见的关键步骤（别人容易踩坑或遗漏）
3. 未来大概率会重复执行类似任务（用户自己说了、或场景明显会复现）
4. 现有技能目录中没有覆盖此模式

**四条全满足才沉淀，否则跳过。**

沉淀时先读取 `skills/skill-creator/SKILL.md` 获取最新规范，SKILL.md 末尾加变更记录区，告知用户。

---

## 环节 3：用户画像

维护一条核心记忆：`tags=["user-profile"]`，`importance=9`，`category="fact"`。

### 画像结构

```
用户画像 - {{用户名}}
角色：{{职业/身份}}
技术栈：{{语言/框架/工具}}
工作风格：{{如"迭代式，全做完再审视两轮"}}
沟通偏好：{{如"简洁直接"}}
决策倾向：{{如"数据驱动" / "先行动后优化"}}
当前关注：{{近期项目/方向}}
已知偏好：{{具体偏好列表}}
```

### 辩证更新法

**先判断是否有新观察。** 本次对话没有暴露新的行为/偏好/模式 → 跳过整个环节 3。

有新观察时，做三步判断防止草率修改：

**步骤 1 — 观察：** 本次对话中看到了什么新行为？

> 例：用户说"一次做完再两轮审视"，与之前记录的"渐进式开发"不同

**步骤 2 — 质疑：** 这是稳定偏好还是特殊情况？

> 例：可能只是对"技能创建"这类任务的做法，编码时仍然渐进式

**步骤 3 — 综合：** 更新画像时保留场景差异

> 例：工作风格更新为"编码渐进式；方法论类任务偏好一次做完+迭代审视"

画像单条内容控制在 500 字以内，超出时精简低价值条目。

---

## 输出规范

有实际产出时，在回复末尾一行简述：

```
---
自进化：记忆 +2（项目配置、技术偏好）| 画像已更新 | 技能沉淀：无
```

三个环节全跳过则不输出。用户催促或不关心时静默执行。沉淀了新技能时必须告知用户。

## 三、数据模型（v2.0 最终版）

### 3.1 模型总览

```
年度O（纯对齐容器，无KR，无进度）
├── Q1 O（季度执行，挂KR，进度=KR加权）
│   ├── Q1 KR1 · 权重50%
│   ├── Q1 KR2 · 权重50%
│   ├── Q1 O(团队A)（对齐，挂KR）
│   └── Q1 O(团队B)（对齐，挂KR）
├── Q2 O
├── Q3 O
└── Q4 O
```

**三条核心规则：**
1. 年度O = 纯对齐容器，不挂KR，不展示进度
2. 季度O = 执行层，挂KR，进度 = 自身KR加权平均
3. O的进度不从子O聚合，树形结构仅承载对齐关系

### 3.2 五张表单

| 表名 | 用途 | 简道云表单类型 |
|------|------|--------------|
| O表 | 年度O + 季度O（同表，周期字段区分） | 普通表单 |
| KR表 | 季度KR，挂载到季度O | 普通表单 |
| 进度更新表 | KR每周进度更新记录 | 普通表单（KR子表） |
| 批示表 | 快速批示记录 | 流程表单 |
| **协作关系表** | KR之间的强协作关系登记 | 普通表单 |

### 3.3 表关系图

```
O表（年度O + 季度O）
 │
 │ 父O ← 关联自身（树形结构）
 │
 ├── KR表（季度O ← 1:N → KR）
 │     │
 │     ├── 进度更新表（KR ← 1:N → 进度更新）
 │     │
 │     └── 协作关系表（KR_A ← N:M → KR_B，双向关联KR表）
 │
 └── 批示表（O/KR ← 1:N → 批示）
```

> 完整字段设计（含计算公式、权重校验、联动逻辑）见 `files/OKR字段设计文档-v2.0.md`

### 4.2 KR协作机制设计（重要决策，v2.1 扩展）

**需求**：当KR负责人发现需要别人强协同时，在KR中登记协作关系，协作人的KR中也明确和该KR有联系。

**讨论过程**：
- 初始方案偏向"发起→审批"流程，被纠正
- 核心定位修正：协作关系表的本质是**表达两个KR之间的强协作关系**，不是驱动流程
- 各部门并行编制O/KR，沟通中发现协作需求，双方达成一致后登记

**v2.0 初始方案（6字段）**：关系编号、KR_A、KR_B、协作说明、登记人、登记时间

**v2.1 扩展方案（12字段，2026-06-08 第二次会话）**：

| # | 字段 | 类型 | 说明 |
|---|------|------|------|
| 1 | 关系编号 | 流水号 | COLLAB-{年份}-{序号} |
| 2 | KR_A | 关联数据→KR表 | 协作一方 |
| 3 | KR_B | 关联数据→KR表 | 协作另一方 |
| 4 | **协作类型** | 单选按钮 | **互锁型 / 依赖型 / 共享交付型** |
| 5 | 协作说明 | 多行文本 | 双方确认的协作内容 |
| 6 | **共享交付物** | 多行文本 | 如有共同交付物 |
| 7 | **协作状态** | 单选按钮 | **活跃 / 已完成 / 已解除** |
| 8 | 登记人 | 成员单选 | 任一方均可登记 |
| 9 | 登记时间 | 日期时间 | 自动 |
| 10 | **对方确认人** | 成员单选 | 对方KR的协作对接人 |
| 11 | **关联季度** | 计算字段 | 从KR带出，用于过滤 |
| 12 | **备注** | 多行文本 | 补充信息 |

**三种协作类型**：
- **互锁型**：双向依赖，A影响B、B也影响A（如：对账差错率 ↔ 对账模块上线）
- **依赖型**：单向依赖，A是B的前置条件（如：API开发 → 数据迁移）
- **共享交付型**：共同产出同一交付物（如：绩效方案 ↔ 薪酬规则）

**业务规则**：
- 不允许自关联、不允许重复关系、同季度约束、单KR最多5条协作
- 状态流转：活跃 → 已完成/已解除（不可逆）
- 解除需填写原因

**通知规则**：
- 新登记 → 通知对方KR责任人
- 状态变更 → 通知双方
- 任一方KR落后 → 通知对方（风险联动）

**KR详情页展示**：
- Tab区新增"协作关系"Tab，带数量角标
- 卡片式展示：对方KR信息 + 进度 + 状态 + 协作说明 + 共享交付物
- 风险联动：对方KR落后时显示预警条
- 支持在线登记（模态框表单）
- 历史协作关系折叠展示

**关键设计点**：
- 普通表单，无流程，无审批
- 两个KR是平等的，不区分"发起方"和"协作方"
- KR详情页双向查询：`WHERE (KR_A=当前KR OR KR_B=当前KR) AND 协作状态="活跃"`
- 前提：双方KR都已存在（并行编制场景下自然满足）

### 4.3 权重语义（已定论）

权重 = 每条子目标对上级目标的贡献度。从根到叶，每层权重加起来 = 100%，进度逐层加权向上汇聚。部门O、团队O、个人O 都在同一张表里填，权重定义从上到下完全一致。

## 五、OGSMA 战略解码框架

| 层 | 含义 | 参与执行？ | 说明 |
|----|------|-----------|------|
| O（目的） | 长期战略方向 | 否 | 一切的方向 |
| G（目标） | 可量化的阶段性结果 | 否 | G 是跨部门合力，不能挂在单一部门名下 |
| S（策略） | 达成 G 的具体路径 | 是 | **被部门O通过「战略来源」领走** |
| M（衡量） | 考核指标定义 | 否 | 注入考核模板 |
| A（责任） | 考核归属 | 否 | S.责任人是HR圈定考核组的锚点 |

**关键决策**：部门领的是 S（策略）而不是 G（目标），因为 G 天然跨部门，S 天然归属单一部门。

## 六、关键业务规则汇总

| 规则 | 实现方式 |
|------|---------|
| 年度O不挂KR | KR表「所属O」关联时，通过数据过滤仅显示周期≠年度的O |
| KR权重之和=100 | 聚合表 + 数据校验 |
| 进度只增不减 | 进度更新表提交时，校验更新后 > 更新前 |
| 每周更新一次 | 进度更新表，同一KR同一周内只能提交一次 |
| O进度=KR加权 | O表计算字段自动聚合 |
| 偏差三档 | 计算字段：正常(≥-10) / 关注(-20~-10) / 落后(<-20) |
| 累计完成值写回KR | 智能助手Pro：进度更新表提交 → 更新KR表 |
| 树形对齐 | O表「父O」字段自关联 |
| KR协作关系登记 | 协作关系表，普通表单，任一方登记 |
| 协作双向可见 | WHERE KR_A=当前KR OR KR_B=当前KR |

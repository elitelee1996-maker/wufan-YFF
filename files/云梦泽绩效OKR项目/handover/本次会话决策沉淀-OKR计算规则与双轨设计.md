## 一、核心架构决策：绩效与OKR双轨独立

### 1.1 结论

绩效管理和OKR管理是**完全独立的两条轨道**，各自有独立的计算逻辑、独立的页面入口、独立的数据流。

```
┌──────────────────────────────────────────────────────┐
│                   员工张明                             │
│                                                      │
│  ┌─────────────────┐    ┌─────────────────┐         │
│  │  绩效轨道(KPI)   │    │  OKR轨道        │         │
│  │                  │    │                 │         │
│  │  绩效指标1: 营收  │    │  O1: 成为华东    │         │
│  │  目标值5000万    │    │  top sales      │         │
│  │  实际完成4800万  │    │  KR1: 签约额     │         │
│  │  完成率96%      │    │  5000万(完成60%) │         │
│  │                 │    │  KR2: 客户满意度  │         │
│  │  绩效指标2: 回款率│    │  ≥90(实际95)    │         │
│  │  目标值≥85%     │    │                 │         │
│  │  实际完成88%    │    │  O2: 建立标准化   │         │
│  │  完成率100%     │    │  销售方法论      │         │
│  │                 │    │  KR1: SOP文档    │         │
│  │  ────────────── │    │  3份(完成2份)    │         │
│  │  绩效总分: 98%  │    │  KR2: 培训覆盖   │         │
│  │  等级: A        │    │  100%(完成80%)   │         │
│  │  → 挂钩奖金/晋升│    │                 │         │
│  │                 │    │  OKR总分: 0.70   │         │
│  │                 │    │  → 复盘用,不挂钩 │         │
│  └─────────────────┘    └─────────────────┘         │
└──────────────────────────────────────────────────────┘
```

### 1.2 两条轨道的定位

| 维度 | 绩效轨道(KPI) | OKR轨道 |
|------|-------------|---------|
| 本质 | 底线指标，必须完成 | 挑战性目标，牵引成长 |
| 和薪酬关系 | 直接挂钩 | 不挂钩 |
| 目标设定 | 可达成的目标 | 挑战性目标，完成60-70%算成功 |
| 打分方式 | 完成率即得分 | 过程用完成率，期末可手动评分 |
| 上下级关系 | 拆解（自上而下分配） | 对齐（方向一致，各自独立） |
| 进度计算 | 逐层加权汇总 | 每个O只由自己的KR决定 |

### 1.3 关联方式

暂时按**没有关联**设计。绩效和OKR之间不做任何数据互通。

## 二、OGSM战略解码地图的定位

### 2.1 OGSM承担「拆解和权重」功能

OGSM战略解码地图是整个系统的战略骨架，负责自上而下的目标拆解和权重分配：

```
O-目的：建成能源化工行业最具公信力的智慧交易平台
├── G-目标：平台交易规模突破1万亿（30%）
│   ├── S-策略：核心品类深度运营（50%）→ 销售管理部
│   └── S-策略：交易模式多样化（50%）→ 销售管理部
├── G-目标：外部客户交易占比突破40%（30%）
│   ├── S-策略：...（50%）→ 生态拓展部
│   └── S-策略：...（50%）→ 生态拓展部
├── G-目标：平台功能达到商用成熟度（25%）
│   └── ... → 技术研发部
└── G-目标：通过等保三级认证（15%）
    └── ... → 安全管理处、财务管理处
```

**权重规则**：
- 每个G-目标的权重是对O-目的的贡献比例，所有G权重合计=100%
- 每个S-策略的权重是对所属G-目标的贡献比例，同一G下所有S权重合计=100%

### 2.2 S-策略层是OKR和KPI的分叉入口

在S-策略层，有两个操作按钮：
- **[制定目标(OKR)]** → 进入OKR轨道，将该策略转化为OKR目标
- **[制定考核指标]** → 进入KPI轨道，将该策略转化为绩效考核指标

```
S-策略: 核心品类深度运营
├── [制定目标(OKR)] → 部门O: 华东签约额行业第一
│                       ├── KR1: 新签商户200家
│                       └── KR2: 续约率≥85%
│
└── [制定考核指标] → 绩效指标: 华东签约额完成率
                      目标值: 2.5亿
                      权重: 30%
```

## 三、OKR轨道：纯对齐模式（飞书模式）

### 3.1 O和O之间是「对齐」关系，不是「拆解」关系

**这是本次会话最重要的决策变更。**

之前的设计是拆解模式（部门O拆解为团队O，进度逐层汇总）。经过讨论，确认改为**飞书OKR的纯对齐模式**：

| 维度 | 拆解模式（已废弃） | 对齐模式（当前方案） |
|------|------------------|-------------------|
| 下级O来源 | 上级分配 | 下级自主制定 |
| 上下关系 | 数字必须加总=上级目标 | 方向一致，数字不必对应 |
| 计算关系 | 下级完成率汇入上级 | 各自独立打分 |
| 权重 | 有，用于进度汇聚 | **去掉**，不做进度计算 |
| 本质 | 命令分解 | 方向共识 |

### 3.2 OKR目标表字段变更

| 字段 | 变更 |
|------|------|
| 目标权重 | **去掉**（或改为优先级标签P0/P1/P2，不参与计算） |
| 权重求和 | **去掉** |
| 对齐上级 | 保留，但改为**纯引用**（不触发任何计算） |
| 目标类型 | 保留（部门级/团队级/个人级） |
| 所属组织 | 保留 |

### 3.3 每个O的进度只由自己的KR决定

```
部门O: 上线平台统一结算中心
├── KR1: 结算中心上线时间 ≤ Q2     完成率100%  权重40%
├── KR2: 接入银行数量 ≥ 5家        完成率80%   权重35%
└── KR3: 日均结算金额 ≥ 1亿        完成率70%   权重25%

→ O进度 = (100×40 + 80×35 + 70×25) / (40+35+25) = 85.5%

团队O: 银行对接与资金监管体系（对齐到上面的部门O）
├── KR1: 完成5家银行API对接        完成率80%   权重50%
├── KR2: 资金监控系统上线          完成率100%  权重30%
└── KR3: 合规报告自动化率≥80%      完成率60%   权重20%

→ O进度 = (80×50 + 100×30 + 60×20) / (50+30+20) = 82%

两个O的进度各自独立，互不影响。对齐关系只在"对齐视图"中展示方向一致性。
```

### 3.4 管理者怎么看整体进度？

通过**OKR仪表盘**做聚合展示，但不级联到O的进度字段：

```
OKR仪表盘（给部门负责人看的）
├── 我的O进度: 85.5%（自身KR算出来的）
├── 对齐我的O的下级O们:
│   ├── 资金管理组O: 82%
│   └── 结算运营组O: 88%
├── 平均进度: 85%（展示用，不回写到任何O）
└── 风险标记: 结算运营组O进度偏低，关注
```

## 四、KR计算规则

### 4.1 KR完成率（过程跟进用，自动计算）

```
KR完成率 = (当前值 - 起始值) / (目标值 - 起始值) × 100%
封顶100%（超额完成不额外加分）
```

### 4.2 KR评分（期末复盘用，手动打分）

```
KR评分: 0.0-1.0，由负责人手动填写
可以和完成率不同（考虑外部因素、目标挑战性）
评分说明: 多行文本，记录评分依据
```

### 4.3 O的得分

```
过程跟进阶段（自动）:
O实时进度 = Σ(KR完成率 × KR权重) / Σ(KR权重)
  KR完成率封顶100%

期末复盘阶段（手动+自动）:
O最终得分 = Σ(KR评分 × KR权重) / Σ(KR权重)
  KR评分由负责人手动填写
```

### 4.4 KR权重保留

同一个O下的KR可以有不同的权重，因为一个O通常有2-4个KR，重要性确实不同。这个权重只在O内部用，不跨O级联。

### 4.5 KR类型

| 类型 | 示例 | 起始值 | 目标值 | 完成率计算 |
|------|------|--------|--------|-----------|
| 数值型 | 净回款额5000万 | 0 | 5000万 | (当前-起始)/(目标-起始) |
| 百分比型 | 客户满意度≥90分 | 70分 | 90分 | (当前-起始)/(目标-起始) |
| 是否型 | 通过等保三级认证 | 0 | 1 | 0%或100% |
| 里程碑型 | Q2完成方案设计 | 0 | 1 | 0%或100% |

### 4.6 完整数值示例

```
部门O: 上线平台统一结算中心

├── KR1: 结算中心上线时间 ≤ Q2
│   权重: 40%  目标值: Q2  当前值: Q2已上线
│   完成率: 100%  过程得分: 1.0
│   期末评分: 1.0  评分说明: "Q2如期上线"
│
├── KR2: 接入银行数量 ≥ 5家
│   权重: 35%  目标值: 5  当前值: 4
│   完成率: 80%  过程得分: 0.8
│   期末评分: 0.8  评分说明: "第5家银行审批延迟到Q3"
│
├── KR3: 日均结算金额 ≥ 1亿
│   权重: 25%  目标值: 1亿  当前值: 7000万
│   完成率: 70%  过程得分: 0.7
│   期末评分: 0.7  评分说明: "交易量爬坡中，符合预期"

→ O实时进度 = (100×40 + 80×35 + 70×25) / 100 = 85.5%
→ O最终得分 = (1.0×40 + 0.8×35 + 0.7×25) / 100 = 0.855
```

## 六、与移交包设计的差异对照

| 移交包设计 | 本次决策 | 变更原因 |
|-----------|---------|---------|
| O和O之间是自引用拆解树 | O和O之间是对齐关系 | OKR方法论标准做法 |
| O进度 = 下级O加权汇聚 | O进度 = 自身KR加权平均 | 对齐模式下不级联 |
| 权重=对上级的贡献度 | OKR去掉权重，OGSM保留权重 | 避免两套权重体系重复 |
| KR只挂在叶子节点 | 每一层O都有自己的KR | OKR基本定义要求 |
| 进度逐层级联向上 | 每个O独立打分 | 对齐模式核心原则 |
| 上级O进度和下级O有关 | 无直接计算关系 | 对齐≠拆解 |

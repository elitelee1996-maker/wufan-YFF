## 二、预置数据集设计

### 2.1 数据集总览

北森提供 **10个预置数据集**（6个目标管理 + 4个绩效管理），简道云通过 **聚合表 + 数据工厂** 实现。

| 数据集编号 | 数据集名称 | 用途 | 简道云实现方式 | 记录数估算 |
|-----------|-----------|------|--------------|-----------|
| TM-01 | 01员工目标数据集 | 员工目标明细查询 | 聚合表 | 员工数 × 周期数 |
| TM-02 | 02 OKR2.0数据集 | OKR明细查询 | 聚合表 | 员工数 × 周期数 × KR数 |
| TM-03 | 03员工目标数据集FOR数据看板 | 看板专用（含计算字段） | 数据工厂 | 部门数 × 周期数 |
| TM-04 | 04 OKR2.0数据集FOR数据看板 | OKR看板专用 | 数据工厂 | 部门数 × 周期数 |
| TM-05 | 05目标&OKR看板制定分析数据集 | 制定阶段分析 | 数据工厂 | 部门数 × 周期数 |
| TM-06 | 06组织目标数据集 | 组织目标汇总 | 聚合表 | 组织数 × 周期数 |
| PM-01 | 01绩效V5 | 绩效结果明细（员工视角） | 聚合表 | 员工数 × 周期数 |
| PM-02 | 02绩效V5-HR专用 | HR视角（含敏感信息） | 聚合表 + 权限过滤 | 员工数 × 周期数 |
| PM-03 | 03组织绩效 | 组织绩效结果 | 聚合表 | 组织数 × 周期数 |
| PM-04 | 04绩效改进 | 改进计划跟踪 | 聚合表 | 改进计划数 |

## 三、目标管理预置数据集详解

### 3.1 TM-01: 01员工目标数据集

**用途**: 员工目标明细查询，支持按部门/周期/状态筛选

**数据来源**:
- 员工目标主表
- 目标明细子表（展开）
- 员工信息表（关联）
- 部门表（关联）

**聚合表配置**:

```yaml
聚合表名称: 01员工目标数据集
数据源: 员工目标主表

关联配置:
  - 关联表1: 目标明细子表
    关联类型: 一对多展开
    展开字段: item_name, item_weight, target_value, current_value, completion_rate
    
  - 关联表2: 员工信息表
    关联字段: employee_id
    获取字段: employee_name, position, position_level, hire_date
    
  - 关联表3: 部门表
    关联字段: department_id
    获取字段: department_name, parent_department

输出字段:
  - target_id: 目标ID
  - cycle_id: 周期ID
  - employee_id: 员工ID
  - employee_name: 员工姓名
  - department_name: 部门名称
  - target_title: 目标标题
  - item_name: 目标项名称（展开后）
  - item_weight: 目标项权重
  - target_value: 目标值
  - current_value: 当前完成值
  - completion_rate: 完成度%
  - overall_progress: 整体进度%
  - status: 目标状态
  - created_time: 创建时间
  - approved_time: 审批通过时间
```

**使用场景**:
- HR查询某部门所有员工的目标完成情况
- 经理查看下属目标明细
- 导出目标明细Excel进行线下分析

### 3.3 TM-03: 03员工目标数据集FOR数据看板

**用途**: 看板专用数据集，包含分组汇总和计算字段

**数据来源**: TM-01员工目标数据集（通过数据工厂加工）

**数据工厂配置**:

```yaml
数据流名称: 03员工目标数据集FOR数据看板
输入数据: 01员工目标数据集

处理节点:
  1. 分组汇总节点:
    - 分组字段: department_name, cycle_id
    - 汇总指标:
      - 目标总数: COUNT(target_id)
      - 按时提交数: COUNTIF(approved_time <= deadline)
      - 按时完成数: COUNTIF(status = "已完成" AND completion_time <= deadline)
      - 平均完成度: AVG(overall_progress)
      - 高风险目标数: COUNTIF(risk_level = "高风险")
      
  2. 计算字段节点:
    - 按时提交率: 按时提交数 / 目标总数 * 100%
    - 按时完成率: 按时完成数 / 目标总数 * 100%
    - 定量目标占比: SUM(IF(item_type = "定量目标", item_weight, 0)) / SUM(item_weight) * 100%
    
  3. 排序节点:
    - 排序字段: 按时提交率（降序）

输出字段:
  - department_name: 部门名称
  - cycle_id: 周期ID
  - 目标总数
  - 按时提交数
  - 按时提交率
  - 按时完成数
  - 按时完成率
  - 平均完成度
  - 高风险目标数
  - 定量目标占比
```

**使用场景**:
- 仪表盘展示各部门目标制定进度对比
- 识别制定进度落后的部门
- 监控定量目标占比是否符合要求

### 3.4 TM-04: 04 OKR2.0数据集FOR数据看板

**用途**: OKR看板专用数据集，包含信心指数统计和进度分布

**数据来源**: TM-02 OKR2.0数据集（通过数据工厂加工）

**数据工厂配置**:

```yaml
数据流名称: 04 OKR2.0数据集FOR数据看板
输入数据: 02 OKR2.0数据集

处理节点:
  1. 分组汇总节点:
    - 分组字段: department_name, cycle_id, confidence_index
    - 汇总指标:
      - OKR数量: COUNT(okr_id)
      
  2. 追加合并节点:
    - 按进度区间分组:
      - 0-25%: COUNTIF(progress_percentage <= 25)
      - 25-50%: COUNTIF(progress_percentage > 25 AND <= 50)
      - 50-75%: COUNTIF(progress_percentage > 50 AND <= 75)
      - 75-100%: COUNTIF(progress_percentage > 75)
      
  3. 计算字段节点:
    - 低风险OKR占比: COUNTIF(confidence_index = "高信心") / OKR总数 * 100%
    - 高风险OKR占比: COUNTIF(confidence_index = "低信心") / OKR总数 * 100%

输出字段:
  - department_name: 部门名称
  - cycle_id: 周期ID
  - 信心指数分布（高/中/低数量）
  - 进度分布（0-25%/25-50%/50-75%/75-100%数量）
  - 低风险OKR占比
  - 高风险OKR占比
```

**使用场景**:
- 仪表盘展示信心指数饼图
- 监控进度分布直方图
- 识别高风险OKR集中的部门

### 3.5 TM-05: 05目标&OKR看板制定分析数据集

**用途**: 制定阶段分析，监控按时提交率和审批通过率

**数据来源**: 员工目标主表 + OKR主表（通过数据工厂合并）

**数据工厂配置**:

```yaml
数据流名称: 05目标&OKR看板制定分析数据集
输入数据: 员工目标主表, OKR主表

处理节点:
  1. 横向连接节点:
    - 将目标表和OKR表合并为统一视图
    - 添加type字段区分类型（目标/OKR）
    
  2. 分组汇总节点:
    - 分组字段: department_name, cycle_id, type
    - 汇总指标:
      - 应提交人数: COUNT(DISTINCT employee_id)（从员工信息表获取该部门总人数）
      - 已提交数: COUNTIF(status != "草稿")
      - 审批通过数: COUNTIF(status = "已生效")
      - 按时提交数: COUNTIF(status != "草稿" AND created_time <= deadline)
      
  3. 计算字段节点:
    - 提交率: 已提交数 / 应提交人数 * 100%
    - 按时提交率: 按时提交数 / 应提交人数 * 100%
    - 审批通过率: 审批通过数 / 已提交数 * 100%

输出字段:
  - department_name: 部门名称
  - cycle_id: 周期ID
  - type: 类型（目标/OKR）
  - 应提交人数
  - 已提交数
  - 提交率
  - 按时提交数
  - 按时提交率
  - 审批通过数
  - 审批通过率
```

**使用场景**:
- 仪表盘展示各部门制定进度对比
- 识别制定进度落后的部门
- 监控审批通过率评估目标质量

| 图表名称 | 图表类型 | 维度 | 指标 | 说明 |

### 7.2 累计值计算

**场景**: 年度目标拆解为月度目标，累计值跟踪

**数据流配置**:

```yaml

  1. 子表单展开节点:
    - 展开月度拆解子表
    
  2. 分组汇总节点:
    - 分组字段: target_id, month
    - 汇总指标: SUM(monthly_completion_value)

  4. 计算字段节点:
    - 累计完成度 = 累计完成值 / 年度目标值 * 100%

  - target_id
  - month
  - monthly_completion_value
  - cumulative_completion_value
  - cumulative_completion_rate
```

## 八、常见问题与解决方案

### 8.1 数据集刷新延迟

**问题**: 聚合表数据不是实时的，存在延迟

**解决方案**:
- **定时刷新**: 配置智能助手每小时自动刷新聚合表
- **手动刷新**: 在仪表盘页面添加"刷新数据"按钮，触发智能助手
- **实时替代方案**: 对于关键指标，直接使用原始表单查询而非聚合表

### 8.2 大数据量性能问题

**问题**: 员工数超过1000人时，聚合表查询缓慢

**解决方案**:
- **分页查询**: 在仪表盘中启用分页功能
- **预计算**: 通过数据工厂预先计算汇总指标，减少实时计算
- **索引优化**: 为常用筛选字段（如department_id, cycle_id）创建索引
- **数据归档**: 将历史周期数据归档到独立表单，减少主表数据量

### 8.3 权限过滤失效

**问题**: 员工能看到其他部门的数据

**解决方案**:
- **检查权限组配置**: 确保数据权限设置为"仅查看本部门"
- **SQL过滤验证**: 若使用自定义SQL过滤，测试SQL是否正确
- **测试账号验证**: 使用不同角色的测试账号验证权限隔离效果

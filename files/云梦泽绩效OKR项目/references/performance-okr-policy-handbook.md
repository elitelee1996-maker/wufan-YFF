## 第一章：总则

### 1.1 制度目的

本制度旨在建立一套"战略穿透、过程透明、结果公正"的绩效与OKR一体化管理体系，实现以下目标：

1. **战略落地**：将公司战略通过 OGSMA 框架逐层解码为可执行的组织目标与个人目标；
2. **过程可控**：通过 OKR 周/双周进度更新与风险预警机制，实现目标执行的动态可视；
3. **评价公正**：通过多维度评估、校准会议、强制分布等手段，确保绩效评级的客观性与公平性；
4. **持续改进**：将绩效结果与人才培养、薪酬激励、PIP改进计划有机联动，形成管理闭环。

### 1.2 适用范围

本制度适用于公司全体正式员工（含试用期）。实习生、外包人员按另行规定执行。

### 1.3 核心管理周期

| 周期类型 | 时间范围 | 适用对象 | 主要活动 |
|---------|---------|---------|---------|
| **年度周期** | 1月1日 - 12月31日 | 全员 | 年度战略解码、年度目标制定、年度绩效考核 |
| **季度周期** | Q1/Q2/Q3/Q4 | 全员 | 季度OKR制定与复盘、季度进度回顾 |
| **月度/双周周期** | 每月/每两周 | 直线经理+员工 | 1v1辅导、进度检查、风险预警 |

### 1.4 绩效管理哲学

本制度融合两大管理实践：

- **飞书OKR模式**：强调目标的透明对齐、高频协作、自我驱动、风险可视化；
- **北森绩效云模式**：强调OGSMA战略解码、多维度评估、强制分布、完整闭环（校准→申诉→电子签→PIP）。

核心理念：**OKR管过程，绩效评结果；OKR追求挑战与成长，绩效考核追求公平与分配。**

### 1.5 关键术语定义

| 术语 | 定义 |
|------|------|
| **OGSMA** | 战略解码五层框架：目的(Objective) → 目标(Goal) → 策略(Strategy) → 衡量(Measure) → 责任(Accountability) |
| **O (Objective)** | 定性目标，描述"要达成什么"，激励人心、方向明确 |
| **KR (Key Result)** | 关键结果，定量衡量"是否达成了O"，含起始值/当前值/目标值 |
| **KPI** | 关键绩效指标，适用于标准化岗位的定量考核指标 |
| **PBC** | 个人业务承诺，适用于项目制或研发岗位的综合性承诺 |
| **强制分布** | 基于预设比例将绩效结果强制分级（S/A/B/C/D）的机制 |
| **校准会议** | 由HRBP组织的跨部门绩效等级校准，消除评价尺度差异 |
| **PIP** | 绩效改进计划，针对连续低绩效员工的限期改进方案 |
| **对齐 (Alignment)** | 下级目标支撑上级目标的关联关系，含向上对齐和横向对齐 |

## 第二章：组织与角色

### 2.1 角色体系全景

本制度定义四类核心角色，各角色在系统中有明确的功能权限与数据权限。

```
                    ┌─────────────┐
                    │  高管/决策层  │
                    │  (战略Owner) │
                    └──────┬──────┘
                           │ 战略方向、目标审批、结果决策
          ┌────────────────┼────────────────┐
          │                │                │
    ┌─────┴─────┐   ┌─────┴─────┐   ┌─────┴─────┐
    │ HR/绩效管理 │   │  直线经理   │   │   员工     │
    │ (流程Owner)│   │ (团队Owner)│   │ (个人Owner)│
    └─────┬─────┘   └─────┬─────┘   └─────┬─────┘
          │                │                │
          └────────────────┼────────────────┘
                           │
                    ┌─────┴─────┐
                    │  系统管理员  │
                    │  (技术Owner)│
                    └───────────┘
```

### 2.2 角色职责详述

#### 2.2.1 高管/决策层

| 职责项 | 具体内容 | 涉及时段 |
|--------|---------|---------|
| 战略制定 | 主持年度/季度战略解码会议，输出公司级OGSMA | 年度Q4 / 季度末月 |
| 目标审批 | 审批各部门/一级部门负责人的目标草案 | 目标制定期（第1-2周） |
| 进度关注 | 月度查看公司OKR全景图，关注红/黄灯目标 | 每月 |
| 结果审阅 | 审阅公司整体绩效分布报告，批准校准后结果 | 考核期末 |
| 决策应用 | 基于绩效结果做出晋升、调薪、奖金池分配决策 | 考核结束后1个月内 |

#### 2.2.2 HR/绩效管理员

| 职责项 | 具体内容 | 涉及时段 |
|--------|---------|---------|
| 制度维护 | 维护本制度、考核模板、指标库、算分规则 | 持续 |
| 活动发起 | 按时发起季度/年度考核活动，配置模板与参评范围 | 考核启动期 |
| 过程监控 | 每日查看填写率、完成率、逾期率，触发催办 | 全过程 |
| 预警管理 | 识别高风险目标（连续2周红灯），推动直线经理介入 | 执行期 |
| 校准组织 | 组织跨部门绩效校准会议，执行强制分布 | 评估期 |
| 申诉处理 | 受理员工申诉，组织复核，反馈处理结果 | 结果确认期 |
| 数据归档 | 将最终结果归档，同步至薪酬系统 | 考核结束后 |

#### 2.2.3 直线经理

| 职责项 | 具体内容 | 涉及时段 |
|--------|---------|---------|
| 目标分解 | 将公司/上级战略分解为部门/团队目标 | 战略解码后1周内 |
| 对齐审批 | 审核下属目标的对齐关系与挑战度，驳回不符合要求的草案 | 目标制定期 |
| 过程辅导 | 每周/双周与下属进行1v1，关注进度、清除障碍、给予反馈 | 执行期 |
| 绩效初评 | 在考核期末对下属进行多维度打分和评语撰写 | 评估期 |
| 参与校准 | 参加部门/跨部门校准会议，陈述评分依据，接受调整 | 校准期 |
| 绩效面谈 | 向每位下属传达最终结果，回顾亮点与不足，制定下期计划 | 结果发布后1周内 |

#### 2.2.4 员工

| 职责项 | 具体内容 | 涉及时段 |
|--------|---------|---------|
| 目标制定 | 基于上级目标和自身岗位，制定个人OKR/KPI草案 | 目标制定期 |
| 对齐确认 | 确保个人目标向上对齐，必要时主动发起横向对齐 | 目标制定期 |
| 进度更新 | 每周更新KR当前值，如实记录进度与风险 | 每周 |
| 目标复盘 | 季度末进行OKR自评（0-1分），完成四象限复盘 | 季度末 |
| 绩效自评 | 在考核活动中填写自评，提交证明材料和总结 | 评估期 |
| 结果确认 | 查看最终绩效结果，选择同意并签署或发起申诉 | 结果确认期 |

#### 2.2.5 系统管理员

| 职责项 | 具体内容 |
|--------|---------|
| 组织架构维护 | 同步HR系统中的部门、岗位、汇报关系至简道云通讯录 |
| 权限配置 | 按角色配置表单/字段/数据权限，确保数据隔离 |
| 二开接口管理 | 维护简道云OpenAPI凭证，监控API调用状态 |
| 数据备份 | 定期导出绩效数据存档，保障数据安全 |

## 第三章：战略解码阶段

### 3.1 阶段概述

**时段**：年度Q4（10月-11月）/ 季度末月最后两周  
**主导角色**：高管/决策层 + HR  
**核心产出**：公司级OGSMA、部门级目标方向、考核周期定义

### 3.2 战略解码流程

#### 步骤1：战略输入准备（HR主导）

HR在战略解码会议前完成以下准备工作：

1. **创建绩效周期**：在简道云 `perf_cycle` 表中创建新周期，定义周期名称（如"2027年度"）、起止日期、周期类型（年度/季度）。
2. **初始化战略解码模板**：在 `战略解码` 表中预置 OGSMA 五层结构框架。
3. **数据准备**：汇总上一周期公司级OKR完成率、组织绩效分布、外部市场数据，作为战略讨论的输入材料。

#### 步骤2：公司级战略解码会议（高管主导）

会议由CEO或COO主持，VP级以上高管参加，时长通常为1-2天。议程如下：

| 议程 | 内容 | 产出 |
|------|------|------|
| 复盘 | 回顾上一周期战略目标达成情况，识别差距根因 | 复盘纪要 |
| 环境分析 | 讨论外部市场变化、竞争格局、客户需求趋势 | PESTEL/五力分析要点 |
| 战略主题提炼 | 确定公司级2-4个战略主题（Strategic Themes） | 战略主题清单 |
| OGSMA分解 | 对每个战略主题按五层框架拆解 | 公司级OGSMA |

**OGSMA拆解示例**（以某SaaS公司为例）：

| 层级 | 内容 | 示例 |
|------|------|------|
| O-目的 | 为什么要做这件事 | "成为中小企业数字化转型的首选平台" |
| G-目标 | 量化的年度目标 | "ARR突破5000万，付费客户数突破800家" |
| S-策略 | 怎么达成目标 | "加大渠道合作伙伴建设，提升产品PLG转化率" |
| M-衡量 | 用什么指标衡量策略效果 | "渠道贡献收入占比≥30%，自助注册转化率≥15%" |
| A-责任 | 谁对结果负责 | CEO → 增长VP(S1) / 产品VP(S2) |

#### 步骤3：战略解码归档（HR主导）

1. 将会议产出的OGSMA逐条录入简道云 `战略解码` 表，按层级关联（G关联O，S关联G...形成树形结构）。
2. 设置各条目的 `responsible_user`（责任人）和 `status`（状态发布）。
3. 通知全体一级部门负责人：战略已发布，请在X日内完成部门目标承接。

### 3.3 战略解码环节的质量标准

| 检查项 | 标准 | 不合格信号 |
|--------|------|-----------|
| O的挑战性 | 基于当前能力基线，实现概率约50-70% | 100%有把握达成 = 目标过低 |
| G的量化度 | 有明确的数字、时间节点、基准线 | 只有定性描述无数字 |
| S的可执行性 | 有明确动作，不空洞 | "加强""提升""优化"等万能动词堆砌 |
| M的完备性 | 有结果指标+过程指标+预警指标 | 只有结果指标 |
| A的明确性 | 精确到人/角色，不模糊 | "相关部门""大家"等模糊表述 |

## 第四章：目标制定与对齐阶段

### 4.1 阶段概述

**时段**：战略解码完成后2周内（年度：11月-12月；季度：末月最后1周+下季第1周）  
**主导角色**：直线经理 + 员工  
**核心产出**：逐层对齐的OKR/KPI树、已审批的个人目标

### 4.2 目标制定流程

#### 4.2.1 第一步：HR配置考核模板与规则

HR在简道云中完成以下配置后，才能启动目标制定流程：

1. **模板配置**（`考核模板` 表）：
   - 选择模板类型（OKR / KPI / 混合），决定本周期使用的考核框架。
   - 配置考核维度及权重。例如：
     - OKR达成（60%）
     - 价值观/行为（20%）
     - 团队贡献（20%）
   - 配置算分规则（加权求和 / 加权求积 / 自定义公式）。

2. **指标库维护**（`指标库` 表）：
   - 确保指标库中有各岗位适用的标准考核指标。
   - 允许直线经理和员工在制定目标时从指标库中选指标，也可以创建新指标。

3. **发起考核活动**（`绩效考核活动` 表）：
   - 创建考核活动，绑定模板和周期。
   - 在 `考核组` 中指定参与考核的人员范围。
   - 完成配置后点击"启动"，系统自动：
     - 通知所有参评人员"目标制定期已开启"
     - 激活各角色的目标起草权限
     - 生成每个人的考核实例（模板×人员）

#### 4.2.2 第二步：部门目标分解（直线经理）

直线经理收到战略解码通知后，在1周内完成：

1. 阅读公司级OGSMA，理解本部门承接的战略方向和关键策略。
2. 在简道云中创建部门级O（若使用 `组织目标` 表），或直接在 `OKR目标` 表中以部门负责人身份创建。
3. 将部门O**向上对齐**到公司级O或G：
   - 在 `目标对齐` 表中建立 `child_o_id`(部门O) → `parent_o_id`(公司O) 的对齐关系。
   - 对齐关系状态设为"待确认"，等待上级确认。
4. 根据需要，发起**横向对齐**：若本部门目标依赖其他部门的产出，主动向协作部门发起横向对齐请求。

**部门目标分解的质量要求**：

- 部门O必须能找到对应的公司级O/G作为上游支撑——不允许出现"孤儿目标"。
- 每个部门O至少拆出2个KR。
- KR必须满足 SMART 原则：Specific（具体）、Measurable（可衡量）、Achievable（可实现但需努力）、Relevant（与O相关）、Time-bound（有时限）。

#### 4.2.3 第三步：个人目标制定（员工）

员工收到"目标制定期已开启"通知后，在1周内完成：

1. **理解上级目标**：查看直线经理的部门O/KR，理解自己需要支撑的方向。
2. **起草个人OKR/KPI**：
   - 在简道云 `OKR目标` 表中创建个人O。
   - 填写 `o_title`（目标标题，定性，激励人心）。
   - 为每个O创建2-5个KR（`关键结果` 表），填写起始值、目标值。
   - 设置O的 `visibility`（可见范围）：默认全员可见（鼓励透明），敏感目标可设"仅自己"或"指定人员"。
3. **建立对齐关系**：
   - 在 `目标对齐` 表中将个人O向上对齐到直线经理的部门O。
   - 若个人O与跨部门同事的O有协作依赖，发起横向对齐。
4. **提交审批**：点击"提交审批"，将OKR草案推送给直线经理。

#### 4.2.4 第四步：目标审批与对齐确认（直线经理）

直线经理收到下属提交的OKR草案后，逐条审核：

| 审核维度 | 检查内容 | 典型驳回原因 |
|---------|---------|------------|
| 对齐度 | 是否有效支撑上级O？能否在对齐视图中看到清晰的支撑链？ | 目标与上级O无关，或支撑关系不成立 |
| 挑战度 | KR目标值是否具有挑战性（50-70%把握达成）？ | KR目标值过低（几乎100%可达成） |
| 完成度 | 是否每个O至少有2个KR？KR是否满足SMART？ | KR模糊无法量化、KR缺失 |
| 权重合理性 | O之间的权重分配是否合理？KR权重之和是否为100%？ | 权重失衡（关键目标权重过低） |

处理结果：
- **审批通过**：对齐关系状态更新为"已确认"，目标状态更新为"已发布"，触发消息通知员工。
- **驳回修改**：填写驳回意见，目标退回员工重新起草。员工修改后重新提交，此过程最多允许2次驳回。

#### 4.2.5 第五步：目标发布与电子签署

所有目标审批通过后：

1. 系统自动将目标状态设为"已发布"。
2. 对于年度目标，系统触发电子签署流程（`签署记录` 表），要求员工和直线经理双方确认签署。
3. 签署完成后的目标进入"冻结期"——原则上不允许修改O和KR结构，仅允许更新KR的 `current_value`。

**目标变更例外**：若因外部环境重大变化（如战略方向调整、组织架构变动）需要调整目标，由直线经理发起"目标变更申请"，经HRBP审批后可解锁修改。

### 4.3 目标对齐全景图（系统产出）

目标制定期结束后，系统应产出以下视图：

- **个人OKR视图**：每位员工可查看自己当前周期的完整OKR及进度。
- **团队OKR看板**：直线经理可查看团队所有成员的OKR及对齐关系。
- **目标地图（Goal Map）**：公司级→部门级→团队级→个人级的四层树形可视化，清晰展示每一条目标支撑链。此功能由二开前端实现（参见 `okr-system-architecture.md`）。

### 4.4 目标制定环节的质量底线

以下任一情况发生时，目标制定期视为不合格，须退回重做：

1. 公司整体目标对齐率低于80%（存在大量"孤儿目标"）。
2. 超过20%的员工未在截止日前提交OKR草案。
3. 超过10%的OKR经审批后发现不符合SMART原则。

## 第五章：过程跟进阶段

### 5.1 阶段概述

**时段**：目标发布后至考核期末的整个执行期  
**主导角色**：员工（进度更新）+ 直线经理（辅导）+ HR（监控预警）  
**核心产出**：实时进度数据、进度趋势图、1v1辅导记录、风险预警

### 5.2 日常进度更新机制

#### 5.2.1 员工端：每周KR更新

员工每周至少更新一次每个KR的 `current_value`（当前值）：

1. 登录简道云，进入个人OKR页面。
2. 逐个KR更新当前值，系统自动计算：
   - `progress_percent` = (current_value - start_value) / (target_value - start_value) × 100%
   - O的 `risk_status`（风险状态）继承所有KR中最差的状态：
     - 正常（绿灯）：所有KR进度均 ≥ 预期进度
     - 有风险（黄灯）：至少1个KR进度落后预期10%-30%
     - 严重滞后（红灯）：至少1个KR进度落后预期超过30%
3. 填写简短的进展备注（可选，但红灯KR必须填写原因说明）。

**进度预期值计算**：系统根据目标周期自动计算每条KR的"应达进度"。例如Q1目标（1月1日-3月31日），在第6周时预期进度约为50%。

#### 5.2.2 系统端：每日快照与趋势记录

简道云智能助手Pro配置以下自动化：

1. **OKR进度每日快照**（定时触发，每日02:00）：
   - 遍历所有进行中的KR，将当日的 `current_value` 和 `progress_percent` 写入 `进度记录` 表。
   - 用途：生成各KR的进度趋势线（数据工厂 + 仪表盘），支持历史回溯。

2. **目标风险预警推送**（定时触发，每周一09:00）：
   - 筛选 `risk_status` 为"有风险"或"严重滞后"的O。
   - 向O负责人和其直线经理发送飞书/钉钉消息，格式：
     ```
     ⚠️ 目标风险预警
     O：[目标标题]
     当前进度：XX%，落后预期 X 个百分点
     风险等级：🔴 严重滞后 / 🟡 有风险
     请及时更新进度并与经理沟通
     ```

### 5.3 1v1辅导机制

#### 5.3.1 辅导频率要求

| 层级 | 最低频率 | 建议时长 | 记录要求 |
|------|---------|---------|---------|
| 直线经理 → 直接下属 | 每两周1次 | 30分钟 | 简道云记录要点 |
| 二级经理 → 隔级下属 | 每月1次 | 30分钟 | 可选 |
| HRBP → 高风险目标负责人 | 按需 | 30分钟 | 必须记录 |

#### 5.3.2 1v1辅导内容框架

每次1v1应覆盖以下四个象限（飞书OKR四象限模式）：

| 象限 | 内容 | 引导问题 |
|------|------|---------|
| **保持 (Keep)** | 当前做得好的、应继续保持的 | "这个周期你做对了什么？哪些方法有效？" |
| **停止 (Stop)** | 需要停止的低效行为或方向 | "有什么事在浪费你的时间？什么问题反复出现？" |
| **开始 (Start)** | 应该开始做的新事情 | "下一阶段你需要开始做什么？有什么新想法？" |
| **继续 (Continue)** | 正在进行但需要优化的 | "哪些方面你还需要更多资源或支持？" |

#### 5.3.3 辅导记录

直线经理在1v1结束后，在简道云中记录简要纪要（可选功能，建议使用 `OKR复盘` 表的扩展字段或专用辅导记录表）。HRBP可查看辅导覆盖率（完成辅导的次数/应辅导次数）。

### 5.4 HR过程监控

HR/绩效管理员在过程跟进阶段的核心监控指标：

| 监控指标 | 目标值 | 预警阈值 | 动作 |
|---------|--------|---------|------|
| KR周更新率 | ≥90% | <80% | 全员催办提醒 |
| 1v1辅导覆盖率 | ≥85% | <70% | 提醒直线经理 |
| 红灯目标占比 | ≤10% | >20% | 组织专项分析会议 |
| 目标对齐率（持续） | ≥95% | 偏离>5% | 检查是否有目标变更未记录 |

### 5.5 季度OKR复盘

每个季度末（最后1周），在执行期的同时启动复盘：

1. **员工OKR自评**（`OKR复盘` 表）：
   - 对每个O进行0-1分自评。
   - 完成四象限复盘总结（保持/停止/开始/继续）。
   - 标记 `difficulty_factor`（难度系数：正常/挑战/极限）。

2. **直线经理复核**：
   - 查看下属的OKR复盘。
   - 填写 `manager_score`（经理打分）和评语。
   - 此打分作为后续绩效考核的重要参考输入。

**重要区分**：OKR复盘打分（0-1分）与绩效考核打分是两套体系。OKR打分重在自我反思和能力成长，绩效打分重在价值评估和利益分配。两者的关联关系在第六章详述。

## 第六章：绩效评估阶段

### 6.1 阶段概述

**时段**：考核周期结束后1-2周  
**主导角色**：HR（发起）+ 员工（自评）+ 直线经理（初评）+ 同事/下属（互评）  
**核心产出**：多维评估结果、个人绩效总分（校准前）

### 6.2 考核活动启动

HR在考核期末完成以下启动操作：

1. 在 `绩效考核活动` 表中将活动状态更新为"评估中"。
2. 系统自动执行以下动作：
   - 为每位参评员工生成一条 `绩效结果` 记录（状态=待评估）。
   - 为每条考核项生成 `评估记录` 记录（评价人待填写）。
   - 向所有参评人员和评价人发送"绩效考核已启动"通知。
   - 激活评价人对应条目的填写权限。

**评价人指派规则**：

| 评价人角色 | 指派规则 | 权重（默认） |
|-----------|---------|------------|
| 员工自己（自评） | 自动 | 10% |
| 直线经理（上级评） | 自动（按汇报关系） | 60% |
| 同事（同事互评） | 直线经理指定2-3人 | 20% |
| 下属（下属评） | 自动（有下属者） | 10% |

> 权重可根据模板配置调整。对于无下属的员工（IC岗），经理权重提升至70%。

### 6.3 多维度评估流程

#### 6.3.1 第一步：员工自评

员工在收到通知后3个工作日内完成：

1. 打开个人考核实例，逐条查看考核项（由模板生成，自动带出维度/指标/权重）。
2. 对每条考核项填写：
   - 自评得分（按预设评分标度，如1-5分或S/A/B/C/D）。
   - 自评依据：简述达成情况、关键成果、证明材料和不足反思。
   - 可上传附件（数据报表、项目文档、客户反馈等）。
3. 填写整体自评总结（200字以内）。
4. 提交自评，系统自动将状态更新为"待上级评价"。

#### 6.3.2 第二步：上级初评（直线经理）

直线经理在自评提交后5个工作日内完成：

1. 查看每位下属的自评内容和OKR执行期数据（进度趋势、1v1记录、OKR复盘得分）。
2. 对每位下属逐条打分：
   - 参考自评，但独立判断（自评仅作为参考，不作为约束）。
   - 评分依据：OKR达成数据 + 行为观察 + 团队贡献 + 1v1评价记录。
   - 填写评语（强制，不少于50字），说明评分理由和改进建议。
3. 对每位下属给出初步总体评价（总分和建议等级）。
4. 提交初评，系统自动将状态更新为"待互评"。

#### 6.3.3 第三步：同事互评与下属评价

系统在初评提交后自动开放互评和下属评价权限，评价人在3个工作日内完成：

**同事互评规则**：
- 评价维度由模板定义（通常为协作、沟通、专业能力等软性维度）。
- 互评结果**匿名**处理（被评者看不到具体谁给了什么分，只看到汇总结果）。
- 确保每名员工至少有3名同事评价人（不足时由HRBP补充指派）。

**下属评价规则**（仅适用于有直接下属的管理者）：
- 评价维度：领导力、决策质量、团队发展、沟通透明度等。
- 同样匿名处理。
- 下属评价结果作为管理者"团队贡献"维度的参考输入，不单独计分。

### 6.4 评分标度与锚定

为防止评分通胀和尺度差异，系统提供统一的评分锚定指引：

| 等级 | 分数区间 | 行为锚定描述 |
|------|---------|-------------|
| **S（卓越）** | 4.5-5.0 | 大幅超越目标，做出超出岗位预期的贡献，有可复制的创新或突破 |
| **A（优秀）** | 3.8-4.4 | 达成所有目标，部分超越目标，主动承担额外责任 |
| **B（良好）** | 3.0-3.7 | 达成大部分目标，符合岗位要求，偶有超出 |
| **C（待改进）** | 2.0-2.9 | 部分目标未达成，需在特定领域改进，但整体尚可 |
| **D（不合格）** | <2.0 | 关键目标严重未达成，需制定PIP改进计划 |

> 直线经理初评时，**建议S比例不超过团队的15%**（作为后续校准的参考基线）。

### 6.5 评估得分计算

系统根据模板配置的算分规则，自动汇总各维度得分：

```
个人绩效总分 = Σ(维度得分 × 维度权重)

维度得分 = Σ(指标得分 × 指标权重)   [按模板的指标-维度层级]

指标得分 = Σ(评价人给分 × 评价人权重)  [自评10% + 上级60% + 同事20% + 下属10%]
```

计算结果写入 `绩效结果` 表的 `total_score` 字段，同时生成各维度明细分。

**OKR数据与绩效评分的关联**：

| 模板类型 | OKR权重 | 处理方式 |
|---------|--------|---------|
| OKR模板 | 60-80% | OKR复盘得分（经理复核后的 score）直接作为"OKR达成度"维度的主要输入 |
| KPI模板 | 0% | 不使用OKR数据，完全基于KPI指标打分 |
| 混合模板 | 30-50% | OKR复盘得分占一部分权重，KPI指标占另一部分 |

> 这说明OKR复盘质量直接影响绩效考核的科学性——复盘不认真，绩效打分就"缺数据"。

## 第七章：校准与定级阶段

### 7.1 阶段概述

**时段**：绩效评估完成后1周  
**主导角色**：HRBP（组织）+ 直线经理（参与）  
**核心产出**：校准后的绩效等级、强制分布结果

### 7.2 为什么要校准

不同直线经理的评分尺度天然存在差异：
- "严苛型"经理：团队平均分3.0，实际表现可能不错。
- "宽容型"经理：团队平均分4.2，实际表现可能一般。

校准会议的核心目的是**消除尺度偏差，确保跨团队公平**。这个过程对标北森绩效云的"绩效校准"模块。

### 7.3 校准会议流程

#### 7.3.1 会前准备（HRBP负责，会议前3天）

1. 汇总所有初评结果，生成以下材料：
   - **部门绩效分布图**：各部门平均分、最高/最低分、S/A建议比例。
   - **矩阵对比视图**（二开实现）：横轴为绩效总分，纵轴为潜力评价（如有），将员工分布可视化。
   - **异常值清单**：同部门中分数异常偏高或偏低的个案。
   - **历史对比**：与上周期同部门同层级的分布对比。
2. 将校准材料提前发送给所有参会直线经理。

#### 7.3.2 会议议程（时长2-4小时，视团队规模）

| 议程 | 时长 | 内容 | 主持 |
|------|------|------|------|
| 整体回顾 | 15分钟 | 公司整体绩效分布概况、同比变化 | HRBP |
| 部门汇报 | 每部门10-15分钟 | 各直线经理陈述初评依据、团队亮点与问题 | 直线经理 |
| 交叉校准 | 45-60分钟 | 横向对比各部门同职级员工，讨论调整方案 | HRBP |
| 强制分布 | 30分钟 | 基于公司整体分布规则，锁定各等级名单 | HRBP |
| 特殊个案 | 15-30分钟 | 讨论PIP人选、晋升建议、绩效申诉预案 | 全体 |

#### 7.3.3 校准操作规则

1. **校准只调等级，不调分数**：原始分保留在系统中作为过程数据，校准调整的是 `final_rating`（最终等级）。所有调整记录写入 `校准记录` 表。

2. **调整必须有依据**：每位被调级的员工，HRBP必须填写 `adjustment_reason`（调整原因），不能仅凭主观感觉。

3. **同部门同职级优先对比**：先确保部门内排序合理，再做跨部门对比。

4. **禁止"水涨船高"**：不能因为某部门整体表现好就把所有人往上调一级——整体好应当体现在更多S/A名额，而非全员升档。

### 7.4 强制分布规则

校准完成后，基于预设的强制分布比例对各等级人数进行最终锁定。

#### 7.4.1 标准分布规则

| 等级 | 比例 | 含义 | 对应管理动作 |
|------|------|------|------------|
| **S** | ≤10% | 卓越，远超预期 | 优先晋升/高额奖金/人才盘点高潜 |
| **A** | ≤25% | 优秀，超越预期 | 正常晋升通道/标准奖金/关注保留 |
| **B** | 50-55% | 良好，符合预期 | 标准调薪/标准奖金/持续发展 |
| **C** | 10-15% | 待改进，部分未达预期 | 限制调薪/降低奖金/制定改进计划 |
| **D** | ≤5% | 不合格，严重未达预期 | 不调薪/无奖金/启动PIP |

> 比例为公司级总控。各部门的分布由强制分布算法引擎（二开服务）根据部门人数和分数分布自动计算，确保公司级比例达标的同时兼顾部门差异。

#### 7.4.2 强制分布的技术实现

此处超出简道云数据工厂的能力范围（涉及多轮迭代分配和全局约束），由二开服务层实现：

```
二开强制分布算法引擎逻辑（伪代码）：

1. 输入：全员 绩效结果 (含 total_score 和直线经理建议等级)
2. 计算每人 z-score = (个人总分 - 公司平均分) / 标准差
3. 初始等级按 z-score 区间映射：
   - z > 1.5 → S
   - 0.5 < z ≤ 1.5 → A
   - -0.5 < z ≤ 0.5 → B
   - -1.5 < z ≤ -0.5 → C
   - z ≤ -1.5 → D
4. 检查各等级人数是否符合比例要求，不符合则边界微调
5. 标记"边界员工"（正好卡在等级边界 ±0.05 标准差范围内），提交HRBP人工裁定
6. 输出：全员 final_rating + 边界员工清单
```

#### 7.4.3 强制分布的特殊情况处理

| 情况 | 处理方式 |
|------|---------|
| 部门人数<5人 | 该部门不单独强制分布，合并到上级部门或相似职能群组统一分布 |
| 新员工（入职<3个月） | 不参与强制分布，由直线经理独立给出评价等级 |
| 考核期内有长期休假（>1个月） | 经HRBP审批后可不参与强制分布，等级单独评定 |
| 试用期员工 | 走独立试用期考核流程，不纳入全员强制分布 |

## 第八章：结果沟通与应用阶段

### 8.1 阶段概述

**时段**：校准完成后2周内  
**主导角色**：直线经理（面谈）+ 员工（确认）+ 高管（决策）+ HR（归档）  
**核心产出**：绩效面谈记录、员工签署确认、薪酬调整方案、PIP/Promotion名单

### 8.2 绩效面谈

#### 8.2.1 面谈流程与要求

校准完成后，HR发布最终绩效结果给各直线经理（此阶段结果仅经理可见，员工尚不可见）。直线经理在收到结果后5个工作日内，与每位下属完成绩效面谈：

1. **会前准备**（直线经理，面谈前1天）：
   - 回顾下属全周期的OKR进度、1v1记录、自评内容、互评反馈。
   - 提炼3个亮点和2个改进点（具体到行为和事例，不空泛）。
   - 准备下周期的初步目标方向。

2. **面谈结构**（建议45-60分钟）：

| 环节 | 时长 | 内容 |
|------|------|------|
| 回顾 | 10分钟 | 本周期OKR/KPI总体达成情况的客观数据回顾 |
| 亮点 | 10分钟 | 直线经理反馈下属做得好的3个方面（有具体事例） |
| 改进 | 10分钟 | 直线经理反馈需要改进的2个方面（有具体事例+改进建议） |
| 结果 | 5分钟 | 告知最终绩效等级，解释评分依据 |
| 对话 | 10-15分钟 | 员工回应、提问、表达观点 |
| 展望 | 5分钟 | 讨论下周期的目标方向和成长路径 |

3. **面谈记录**：直线经理在面谈结束后24小时内在简道云中填写简要面谈纪要（关键结论+双方共识），作为绩效结果的附加说明。

#### 8.2.2 面谈红线

以下行为属于面谈违规，员工可向HRBP投诉：
- 只在面谈中告知结果不解释原因。
- 拿同事做不当比较（"你看看XXX做得多好"）。
- 只批评不给建设性建议。
- 面谈时间少于15分钟。

### 8.3 结果确认与电子签署

面谈完成后，HR在系统内将绩效结果状态切换为"待确认"，员工端可见。

1. **员工确认**：员工登录系统查看最终绩效结果和直线经理评语。
   - **同意**：点击"确认结果"，触发电子签署流程（`签署记录` 表）。签署完成后结果锁定，状态变为"已确认"。
   - **异议**：点击"发起申诉"，进入第九章申诉程序。

2. **电子签署**：
   - 员工和直线经理双方完成电子签名确认。
   - 签署信息（签名图、时间戳、IP）存入 `签署记录` 表。
   - 签署完成后，绩效结果状态变为"已签署"，数据进入归档状态。

### 8.4 结果应用

绩效结果经签署确认后，HR/高管按以下规则应用：

#### 8.4.1 薪酬调整

| 绩效等级 | 调薪幅度（建议） | 奖金系数 |
|---------|----------------|---------|
| S | 10%-20% | 1.5-2.0x |
| A | 5%-10% | 1.2-1.5x |
| B | 0%-5% | 1.0x |
| C | 0% | 0.5-0.8x |
| D | 0%（且不参与当年调薪） | 0x |

> 具体数值由薪酬委员会根据年度经营情况最终确定，本制度仅给出参考区间。

#### 8.4.2 人才盘点与晋升

- **S等级**：自动进入"高潜人才池"，纳入人才九宫格"高绩效-高潜力"象限，优先考虑晋升。
- **A等级**：纳入"核心人才"关注名单，直线经理应在下周期给予更多挑战性任务。
- **连续2次C等级**：HRBP与直线经理制定定向改进计划（非正式PIP），明确1个季度内的改进目标。
- **D等级或连续2次C等级改进无效**：启动正式PIP计划（见8.5节）。

#### 8.4.3 培训与发展

基于绩效评估中暴露的共性短板，HR在考核结束后1个月内制定组织级培训计划：
- 评分普遍偏低的维度 → 针对性培训课程。
- 管理者"团队发展"维度得分偏低 → 领导力提升专项。
- 新晋升管理者首次评估 → 新经理融入辅导。

### 8.5 PIP绩效改进计划

#### 8.5.1 触发条件

满足以下任一条件，启动正式PIP：

1. 单次绩效考核等级为D。
2. 连续两次绩效考核等级为C。
3. 直线经理和HRBP共同评估认为需要干预的其他情况（需经HR负责人审批）。

#### 8.5.2 PIP流程

| 步骤 | 内容 | 时限 | 负责人 |
|------|------|------|--------|
| 启动 | HRBP与直线经理拟定PIP方案，明确改进目标、行动计划、时间节点、检查频率 | 绩效发布后1周内 | HRBP + 直线经理 |
| 沟通 | 与员工正式沟通PIP内容，签署PIP确认书 | 启动后3天内 | 直线经理 |
| 执行 | 员工按行动计划执行改进，每周向直线经理汇报进展，每两周HR检查 | 通常60-90天 | 员工 + 直线经理 |
| 评估 | PIP到期后，直线经理和HRBP共同评估改进效果 | PIP到期后1周内 | HRBP + 直线经理 |
| 决定 | 改进成功→恢复正常绩效管理流程；改进失败→进入离职协商或调岗程序 | 评估后1周内 | HRBP + 直线经理 |

PIP期间员工不参与当前周期的绩效考核，PIP结果单独评定。

## 第九章：申诉与特别程序

### 9.1 申诉的触发

员工在以下情况下可发起申诉：

1. 认为绩效评估过程中存在程序违规（如评价人指定不当、缺少必要的评价环节）。
2. 认为绩效结果存在事实性错误（如数据引用错误、关键成果被遗漏）。
3. 认为存在不当偏见或歧视影响评估结果。

> 注：单纯对绩效等级不满不构成申诉理由——员工必须指出具体的程序问题或事实错误，而非仅表达"觉得不公平"。

### 9.2 申诉流程

| 步骤 | 内容 | 时限 |
|------|------|------|
| 发起 | 员工在系统内提交申诉申请（`绩效申诉` 表），说明申诉理由并上传证据 | 结果发布后5个工作日内 |
| 受理 | HRBP审核申诉是否符合受理条件，通知员工受理结果 | 收到申诉后3个工作日内 |
| 调查 | HRBP独立调查（查阅评估记录、访谈相关评价人、核验数据） | 受理后5个工作日内 |
| 裁定 | 由HRBP + 隔级经理（申诉人的上级的上级）+ 1名其他部门经理组成三人裁定小组，做出最终裁定 | 调查结束后3个工作日内 |
| 反馈 | HRBP将裁定结果和理由书面反馈给申诉员工和直线经理 | 裁定后2个工作日内 |

**申诉期间的保护**：
- 申诉发起后，该员工的绩效结果状态自动变为"申诉中"，暂停归档和薪酬应用。
- 严禁因员工发起申诉而对其进行任何形式的报复或歧视。
- 申诉处理期间的薪酬暂按原等级执行，待最终裁定后多退少补。

### 9.3 申诉裁定结果

| 裁定结果 | 后续处理 |
|---------|---------|
| 申诉成立（程序问题） | 重新执行评估流程（至少增加1名独立评价人），结果以重评后的为准 |
| 申诉成立（事实错误） | 纠正错误数据，重新计算绩效总分和等级 |
| 申诉不成立 | 维持原结果，结案归档。此裁定为终局裁定，不可再次申诉 |

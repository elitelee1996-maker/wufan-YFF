# 云梦泽 OKR 字段设计文档 v2.0

> 基于模型确认：年度O纯对齐容器 + 季度O挂KR执行 + O进度仅从自身KR聚合

---

## 一、模型总览

```
年度O（纯对齐，无KR，无进度）
├── Q1 O（季度执行，挂KR，进度=KR加权）
│   ├── Q1 KR1 · 权重50%
│   ├── Q1 KR2 · 权重50%
│   ├── Q1 O(团队A)（对齐，挂KR）
│   └── Q1 O(团队B)（对齐，挂KR）
├── Q2 O（季度执行，挂KR）
│   └── ...
├── Q3 O
└── Q4 O
```

**三条核心规则：**
1. 年度O = 纯对齐容器，不挂KR，不展示进度
2. 季度O = 执行层，挂KR，进度 = 自身KR加权平均
3. O的进度不从子O聚合，树形结构仅承载对齐关系

---

## 二、五张表单

| 表名 | 用途 | 简道云表单类型 |
|------|------|--------------|
| O表 | 年度O + 季度O（同表，周期字段区分） | 普通表单 |
| KR表 | 季度KR，挂载到季度O | 普通表单 |
| 进度更新表 | KR每周进度更新记录 | 普通表单（KR子表） |
| 批示表 | 快速批示记录 | 流程表单 |
| 协作关系表 | KR之间的强协作关系登记 | 普通表单 |

## 三、O表字段设计

| # | 字段名称 | 字段类型 | 必填 | 说明 |
|---|---------|---------|------|------|
| 1 | O编号 | 流水号 | 自动 | 规则：O-{部门}-{年份}-{序号} |
| 2 | O名称 | 单行文本 | 是 | 如"提升结算中心运营效率" |
| 3 | O描述 | 多行文本 | 否 | 对O的详细说明 |
| 4 | 周期 | 单选按钮 | 是 | 选项：年度 / Q1 / Q2 / Q3 / Q4 |
| 5 | 父O | 关联数据 | 否 | 关联到O表自身，构建树形结构 |
| 6 | 责任人 | 成员单选 | 是 | O的负责人 |
| 7 | 所属部门 | 部门单选 | 是 | 归属部门/团队 |
| 8 | 开始日期 | 日期 | 是 | 周期的起始日期 |
| 9 | 截止日期 | 日期 | 是 | 周期的截止日期 |
| 10 | KR数量 | 数字(计算) | 自动 | 统计该O下的KR数量 |
| 11 | KR权重合计 | 数字(计算) | 自动 | 该O下所有KR权重之和 |
| 12 | 完成度 | 数字(计算) | 自动 | **季度O**：Σ(KR完成度 × KR权重) / 100；**年度O**：空 |
| 13 | 预期进度 | 数字(计算) | 自动 | 时间均匀计算：(今天-开始日期)/(截止日期-开始日期)×100 |
| 14 | 偏差 | 数字(计算) | 自动 | 完成度 − 预期进度 |
| 15 | 状态 | 单行文本(计算) | 自动 | **年度O**：空；**季度O**：偏差≥−10→正常，−20≤偏差<−10→关注，偏差<−20→落后 |

### 计算字段公式

**完成度（季度O）：**
```
IF(周期="年度", "", ROUND(SUM(KR表.完成度 × KR表.权重) / 100, 0))
```

**预期进度：**
```
IF(周期="年度", "", ROUND((TODAY() - 开始日期) / (截止日期 - 开始日期) * 100, 0))
```

**偏差：**
```
IF(周期="年度", "", 完成度 - 预期进度)
```

**状态：**
```
IF(周期="年度", "", IF(偏差 >= -10, "正常", IF(偏差 >= -20, "关注", "落后")))
```

## 四、KR表字段设计

| # | 字段名称 | 字段类型 | 必填 | 说明 |
|---|---------|---------|------|------|
| 1 | KR编号 | 流水号 | 自动 | 规则：KR-{年份}-{序号} |
| 2 | KR名称 | 单行文本 | 是 | 如"首月对账差错率不超过0.3%" |
| 3 | KR描述 | 多行文本 | 否 | 对KR的详细说明，含量化指标 |
| 4 | 所属O | 关联数据 | 是 | 关联到O表，**仅关联周期=Q1/Q2/Q3/Q4的O** |
| 5 | 权重 | 数字 | 是 | 范围 1-100，同级KR权重之和需等于100 |
| 6 | 目标值 | 单行文本 | 是 | 如"≤0.3%"、"95%"，自由文本描述 |
| 7 | 累计完成值 | 数字 | 是 | **人工每周填写**，范围 0-100。代表"截至目前完成了多少" |
| 8 | 完成度 | 数字(计算) | 自动 | = 累计完成值（直接映射，100分制下完成度=累计完成值） |
| 9 | 开始日期 | 日期 | 是 | KR的起始日期 |
| 10 | 截止日期 | 日期 | 是 | KR的截止日期 |
| 11 | 责任人 | 成员单选 | 是 | KR的执行负责人 |
| 12 | 预期进度 | 数字(计算) | 自动 | 时间均匀：(TODAY()-开始日期)/(截止日期-开始日期)×100 |
| 13 | 偏差 | 数字(计算) | 自动 | 完成度 − 预期进度 |
| 14 | 状态 | 单行文本(计算) | 自动 | 偏差≥−10→正常，−20≤偏差<−10→关注，偏差<−20→落后 |
| 15 | 上次更新时间 | 修改时间 | 自动 | 最后一次修改累计完成值的时间 |

### 计算字段公式

**完成度：**
```
累计完成值
```

**预期进度：**
```
ROUND((TODAY() - 开始日期) / (截止日期 - 开始日期) * 100, 0)
```

**偏差：**
```
完成度 - 预期进度
```

**状态：**
```
IF(偏差 >= -10, "正常", IF(偏差 >= -20, "关注", "落后"))
```

### 权重校验

使用聚合表 + 数据校验实现：
- 聚合表：按「所属O」分组，SUM(权重)
- 数据校验：权重合计 ≠ 100 时提示"KR权重之和必须等于100%"

## 五、进度更新表字段设计

| # | 字段名称 | 字段类型 | 必填 | 说明 |
|---|---------|---------|------|------|
| 1 | 进度更新编号 | 流水号 | 自动 | 唯一标识 |
| 2 | 所属KR | 关联数据 | 是 | 关联到KR表 |
| 3 | 更新人 | 成员单选 | 自动 | 当前用户 |
| 4 | 更新时间 | 日期时间 | 自动 | 提交时间 |
| 5 | 更新前完成值 | 数字 | 自动 | 从KR表带出当前累计完成值 |
| 6 | 更新后完成值 | 数字 | 是 | 用户填写新的累计完成值 (0-100) |
| 7 | 增量 | 数字(计算) | 自动 | 更新后 − 更新前 |
| 8 | 备注 | 多行文本 | 否 | 本周进展说明 |

### 联动逻辑

1. 选择「所属KR」后，自动带出当前累计完成值到「更新前完成值」
2. 提交后，用智能助手将「更新后完成值」写回KR表的「累计完成值」字段
3. 更新后完成值 > 更新前完成值（不允许倒退）

## 七、协作关系表字段设计（v2.1）

> **定位**：表达两个KR之间的强协作关系，不是流程审批，是关系登记。各部门并行编制O/KR，沟通中发现协作需求，双方达成一致后由任一方登记。

### 7.1 字段清单

| # | 字段名称 | 字段类型 | 必填 | 说明 |
|---|---------|---------|------|------|
| 1 | 关系编号 | 流水号 | 自动 | 规则：COLLAB-{年份}-{序号} |
| 2 | KR_A | 关联数据 | 是 | 关联到KR表，协作一方 |
| 3 | KR_B | 关联数据 | 是 | 关联到KR表，协作另一方 |
| 4 | 协作类型 | 单选按钮 | 是 | 互锁型 / 依赖型 / 共享交付型 |
| 5 | 协作说明 | 多行文本 | 是 | 双方确认的协作内容描述 |
| 6 | 共享交付物 | 多行文本 | 否 | 如有共同交付物，描述内容和时间要求 |
| 7 | 协作状态 | 单选按钮 | 是 | 活跃 / 已完成 / 已解除（默认：活跃） |
| 8 | 登记人 | 成员单选 | 自动 | 谁登记的（任一方均可） |
| 9 | 登记时间 | 日期时间 | 自动 | 提交时间 |
| 10 | 对方确认人 | 成员单选 | 否 | 对方KR的协作对接人（可不同于KR责任人） |
| 11 | 关联季度 | 单行文本(计算) | 自动 | 从KR_A带出所属O的周期，用于过滤 |
| 12 | 备注 | 多行文本 | 否 | 补充信息，如风险、变更说明等 |

### 7.2 协作类型定义

| 类型 | 含义 | 典型场景 | 关注重点 |
|------|------|---------|---------|
| **互锁型** | 两个KR互相依赖，A的进展影响B，反之亦然 | 财务处"对账差错率" ↔ IT处"对账模块上线" | 双向进度同步、风险联动 |
| **依赖型** | A是B的前置条件，A完成后B才能推进 | IT处"API接口开发" → 运营处"数据迁移" | 前置条件完成时间、阻塞预警 |
| **共享交付型** | 两个KR共享同一个交付物，需协调产出 | HR"绩效方案" ↔ 财务"薪酬规则" | 交付物质量、时间线对齐 |

### 7.3 业务规则

| 规则 | 实现方式 |
|------|---------|
| 不允许自关联 | KR_A ≠ KR_B，提交时校验 |
| 不允许重复关系 | 同一对KR只能有一条活跃协作关系（A↔B = B↔A） |
| 同季度约束 | KR_A 和 KR_B 必须属于同一季度周期 |
| 数量上限 | 单个KR最多登记 5 条协作关系 |
| 删除权限 | 仅登记人可删除/解除自己登记的协作关系 |
| 状态流转 | 活跃 → 已完成 / 已解除（不可逆） |
| 已完成条件 | 双方KR均达到100%时，系统提示标记为"已完成" |
| 解除需备注 | 将状态改为"已解除"时，必须在备注字段说明原因 |

### 7.4 双向查询逻辑

KR详情页通过关联查询展示协作关系，无需区分方向：

```
查询条件：协作关系表 WHERE (KR_A = 当前KR OR KR_B = 当前KR) AND 协作状态 = "活跃"
```

已解除/已完成的协作关系在"历史记录"子区域可查看，不在主列表显示。

### 7.5 通知规则

| 触发事件 | 通知对象 | 通知内容 |
|---------|---------|---------|
| 新登记协作关系 | KR_B的责任人 + 对方确认人 | "XX登记了与你的KR的协作关系，请查看" |
| 协作状态变更 | 双方KR责任人 | "协作关系COLLAB-XX状态变更为XX" |
| 任一方KR状态变为"落后" | 对方KR责任人 | "协作方KR-XX进度落后，请关注影响" |

## 八、表关系图

```
O表（年度O + 季度O）
 │
 │ 父O ← 关联自身（树形结构）
 │
 ├── KR表（季度O ← 1:N → KR）
 │     │
 │     ├── 进度更新表（KR ← 1:N → 进度更新）
 │     │
 │     └── 协作关系表（KR_A ← N:M → KR_B，双向关联KR表）
 │
 └── 批示表（O/KR ← 1:N → 批示）
```

## 九、关键业务规则汇总

| 规则 | 实现方式 |
|------|---------|
| 年度O不挂KR | KR表「所属O」关联时，通过数据过滤仅显示周期≠年度的O |
| KR权重之和=100 | 聚合表 + 数据校验 |
| 进度只增不减 | 进度更新表提交时，校验更新后 > 更新前 |
| 每周更新一次 | 进度更新表，同一KR同一周内只能提交一次（通过日期校验） |
| O进度=KR加权 | O表计算字段自动聚合 |
| 偏差三档 | 计算字段，无需人工判断 |
| 累计完成值写回KR | 智能助手Pro：进度更新表提交 → 更新KR表对应字段 |
| 树形对齐 | O表「父O」字段自关联 |
| KR协作关系登记 | 协作关系表，普通表单，任一方登记，双方沟通一致后提交 |
| 协作双向可见 | KR详情页查询：WHERE KR_A=当前KR OR KR_B=当前KR |

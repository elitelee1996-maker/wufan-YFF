# knowledge-retriever 委派指令模板（🆕 v2.2 串行流程 + 产出容错）

## 执行时机

**阶段2**：requirement-analyst 完成 Sheet 1 后，主 Agent 提取 L3 节点列表，委派 knowledge-retriever 做精准检索。

**不再与阶段1 并行**。原因：用具体的 L3 节点（如「CRM 合同触发项目建档」「外购设备关键生产节点」）做检索，匹配精度远高于阶段0 的模糊关键词。

## 输入

主 Agent 从 Sheet 1 提取的节点列表：

| 字段 | 说明 |
|------|------|
| 编号 | Sheet 1 编号 |
| 功能模块 | L1+L2 |
| 功能名称 | L3 流程名 |
| 需求说明 | L4 展开描述 |
| 预估人天 | 标准配置 + 接口/开发 |

## 委派指令结构

```
请基于以下 L3 节点列表，逐项检索历史案例和产品能力边界：

## 项目信息
- 客户名称：[客户名]
- 项目名称：[项目名]
- 行业：[行业]
- 涉及系统：[CRM/PLM/SAP/SRM/WMS/质量/OA/...]

## L3 节点列表
[从 Sheet 1 提取的节点表格：编号/功能模块/功能名称/需求说明]

## 检索要求

### 双路检索
1. **历史案例匹配**：用 `knowhow_search` 检索每个节点是否有同类案例
   - 匹配到案例时：记录案例名称、预估人天、实际人天、偏差率
   - 未匹配到时：标注「无历史案例」
2. **产品能力边界**：用 `jdy-custom-dev-capabilities` 技能检查每个节点
   - 标准品是否支持？
   - 是否需要二开？二开模式是什么？
   - 是否有插件/自定义详情页/AI 功能可替代？
   - 涉及的硬性约束（H1-H11）是否触发？
3. **权限风险**：用 `jdy-permission-model` 技能检查二开节点的权限断裂点

### 风险识别
基于检索结果，识别以下风险并给出草稿：
- 集成风险：多系统接口范围不清、字段未定
- 范围风险：颗粒度下沉可能导致人天膨胀
- 协同风险：供应商入口冲突等
- 附件风险：大文件/多版本边界
- 报表风险：看板指标数据来源
- 计划风险：变更重算规则
- 权限风险：复杂权限矩阵
- 数据风险：历史数据迁移

## 输出要求
返回**结构化数据**（JSON/表格），不产出独立 Markdown 报告文件。结构如下：

### 1. 逐节点检索结果表
| 编号 | 功能名称 | 历史案例数 | 最佳匹配案例 | 案例人天 | 偏差率 | 产品支持度 | 二开需求 | 风险等级 | 备注 |

### 2. 风险清单草稿
| 序号 | 风险分类 | 风险项 | 风险说明 | 风险等级 | 建议应对措施 | 责任方 | 确认节点 |

### 3. 整体评估
- 行业匹配度：高/中/低
- 综合偏差率：X%
- 建议成本缓冲：X%
- 关键待确认事项：逐条列出
```

## 委派后主 Agent 动作（🆕 v2.2 更新）

1. 读取 knowledge-retriever 返回的结构化数据
2. 将逐节点检索结果回填到 Sheet 1（在报价理由列补充匹配度和偏差率）
3. 将风险清单草稿填入 Sheet 3
4. 完成 Sheet 2（项目评估报价）—— 从 Sheet 1 汇总人天
5. 完成 Sheet 4（运维服务模板）
6. 生成 HTML 需求分析报告
7. 弹出合并确认卡片

## 🆕 v2.2 产出容错（2026-06-15 康尼机电复盘新增）

### 如果子 Agent 返回的检索结果不完整（缺少逐节点数据 / 风险清单为空）
主 Agent 不等待重新委派，直接基于以下来源自行补充：
- 阶段0 预标注的风险点
- `cost-estimation-model` 技能的人天基准
- `jdy-custom-dev-capabilities` 技能的能力边界
- `jdy-architecture-patterns` 技能的范式判定

**关键**：不要再次委派同一个子 Agent 做同样的事。一次失败后主 Agent 自行补齐。

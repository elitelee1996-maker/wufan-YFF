# 简道云产品更新跟踪方案

> 基于 `hc.jiandaoyun.com/update` 页面结构研究，为擎析评估系统设计的产品知识保鲜机制。

---

## 一、页面结构研究结论

### 1.1 数据源分析

| 维度 | 发现 |
|------|------|
| **主入口** | `https://hc.jiandaoyun.com/update` |
| **详情页** | `https://hc.jiandaoyun.com/doc/{doc_id}`（doc_id 为递增数字） |
| **侧边栏** | 完整年代索引（2018-2026），格式：`YYYY-MM-DD更新日志 → /doc/{id}` |
| **更新频率** | 约每 2-3 周一次版本发布 |
| **RSS/API** | ❌ 不存在公开的 RSS 或 API 端点 |

### 1.2 版本分类体系

更新日志按以下模块分类：

- **通用功能**：表单、分享、导出等基础能力
- **业务流程**：审批流、流程节点、会签/或签
- **表单管理**：子表单、字段类型、数据关联
- **数据分析**：仪表盘、数据工厂、聚合表
- **开放平台**：API、Webhook、第三方集成
- **组织管理**：部门、角色、权限
- **飞书集成模式**：飞书消息、审批对接
- **知识库**：AI问答、文档管理

### 1.3 补充信息源

| 来源 | URL | 用途 |
|------|-----|------|
| 产品公告 | `/doc/15271` | 重大功能发布预告 |
| 产品月报 | `/doc/15272` | 月度更新汇总 |

## 二、三层跟踪机制

### Layer 1：评估前自动校验（每次评估触发）

**触发时机**：擎析阶段 0 前置洞察

**执行步骤**：

```
1. 读取 files/jdy_capability_baseline.json
   → 获取 latest_version（如 V11.2.0）

2. web_fetch("https://hc.jiandaoyun.com/update")
   → 提取页面中实际最新版本号

3. 比对：
   ├── 一致 → ✅ 知识基线有效，正常继续
   └── 不一致 →
       🟡 P1 告警：
       "简道云已更新至 V{x.y.z}（基线记录为 V{a.b.c}）
        以下能力判断可能受影响：
        {从 skill_mapping 提取的相关技能书列表}
        建议人工复核或执行 Layer 2 扫描"
```

**优势**：零额外成本，嵌入现有流程。

---

### Layer 2：定期变更扫描（每月/按需触发）

**触发时机**：用户主动要求，或每月定期

**执行步骤**：

```
1. 从 baseline.json 获取最新 3 个 doc_id

2. web_fetch 对应的 /doc/{id} 详情页
   → 提取所有变更条目，按模块分类

3. 读取 6 个技能书的能力清单：
   - jdy-architecture-patterns: 架构范式是否受影响？
   - jdy-custom-dev-capabilities: 二开边界是否调整？
   - jdy-api-handbook: API 是否有新增/废弃/行为变更？
   - ipd-standardization: 需求拆解框架需补充新模块？
   - cost-estimation-model: 人天基准需调整？
   - jdy-hybrid-sop: SOP 流程需适配新功能？

4. 输出《变更影响分析报告》
   → 标注每个变更对技能书的影响级别（P0/P1/P2）
```

**产出**：`变更影响分析报告_{YYYY-MM}.md`

---

### Layer 3：深度知识同步（大版本更新时触发）

**触发时机**：

- major 版本号变化（如 V11→V12）
- 或累积 3+ 次 Layer 1 告警未处理

**执行步骤**：

```
1. 全量抓取最近半年的所有更新日志
   → 从 sidebar_index 获取所有 doc_id
   → 逐个 web_fetch 提取变更内容

2. 结构化解析所有变更条目
   → 按模块分类
   → 识别新增能力、废弃能力、行为变更

3. 更新对应技能书：
   → 修改 SKILL.md 中的能力描述
   → 更新反例库（如有新的坑）
   → 更新子 Agent 提示词中的能力边界

4. 重建基线：
   → 更新 jdy_capability_baseline.json
   → 更新 latest_version 和 known_versions

5. 输出《知识同步报告》
```

**产出**：

- 更新后的技能书 SKILL.md
- `知识同步报告_{YYYY-MM-DD}.md`
- 更新后的 `jdy_capability_baseline.json`

## 三、与擎析的集成点

| 擎析阶段 | 集成方式 | 具体动作 |
|---------|---------|---------|
| **阶段 0 前置洞察** | Layer 1 自动校验 | 比对最新版本号，发现差异则 P1 告警 |
| **阶段 2 经验检索** | Layer 2 补充检索 | knowledge-retriever 额外检索最新变更 |
| **阶段 3 架构设计** | 能力边界校验 | architect 参考最新变更判断可行性 |
| **阶段 6 复盘沉淀** | Layer 3 知识回写 | 将项目中发现的产品能力变化写入技能书 |

## 四、使用指南

### 4.1 日常使用（评估前）

当你启动擎析评估新项目时，阶段 0 会自动执行 Layer 1 检查：

```
你：评估一个新项目...
擎析：（自动执行 Layer 1）
      ✅ 产品知识基线有效（当前版本 V11.2.0）
      或
      🟡 P1 告警：简道云已更新至 V11.3.0，建议执行 Layer 2 扫描
```

### 4.2 月度扫描

每月初或产品大更新后，主动触发 Layer 2：

```
你：做一次产品更新扫描
擎析：（执行 Layer 2）
      → 抓取最新 3 个版本的详细变更
      → 与技能书交叉比对
      → 输出《变更影响分析报告》
```

### 4.3 大版本更新

当 major 版本号变化时，触发 Layer 3：

```
你：简道云出了 V12，做深度同步
擎析：（执行 Layer 3）
      → 全量解析最近半年更新
      → 更新技能书 + 反例库
      → 重建基线文件
```

## 六、注意事项

1. **基线文件需手动更新**：Layer 2/3 执行后，需将结果写回 `jdy_capability_baseline.json`
2. **web_fetch 可能失败**：如页面结构变化，需重新研究页面结构
3. **技能书更新需人工审核**：Layer 3 自动生成的技能书修改建议，需人工确认后再应用
4. **无 RSS/API**：简道云更新日志没有公开的 RSS 或 API，只能通过网页抓取

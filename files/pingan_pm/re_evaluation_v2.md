# 平安快运 PMS · 范式F重评估报告 V2.0

> **评估日期**: 2026-06-14
> **评估依据**: 范式F（双模式协同范式）+ T8（双模式触发模式）+ 简道云表单双重角色认知
> **对比基线**: V1.0 评估（范式A+C+L4，120人天中值）

---

## 一、核心变化：范式F带来的3个认知翻转

| # | V1.0 认知 | V2.0 认知（范式F） | 影响 |
|---|----------|-------------------|------|
| 1 | 简道云表单 = 数据库 + 简单CRUD界面 | 简道云表单 = **数据底座 + 交互前端闭环** | 单条记录编辑不再二开，直接用原生详情页 |
| 2 | 二开页面 = 全部自定义（含CRUD） | 二开页面 = **只做"面"（聚合+计算+可视化）** | 删除通用CRUD后端，减少5-8人天 |
| 3 | 智能助手 = 通知推送 | 智能助手 = **数据变更→Bridge回调→重算→写回** | 形成双向闭环，不是单向通知 |

## 三、架构调整

### 3.1 范式判定变更

| 维度 | V1.0 | V2.0 |
|------|------|------|
| **范式** | A + C + L4 | **A + F + L4** |
| **简道云角色** | 数据库 + 简单表单 | **数据底座 + 交互前端闭环** |
| **Bridge层职责** | API代理 + 计算引擎 + CRUD | **API代理 + 计算引擎 + Webhook接收**（不做CRUD） |
| **数据流** | 单向：二开→API→简道云 | **双向闭环：汇聚页⇄原生详情页⇄Bridge** |

### 3.2 Bridge层职责精简

| 服务 | V1.0职责 | V2.0职责 | 变化 |
|------|---------|---------|------|
| JDY Client | API转发+缓存+分页 | API转发+缓存+分页 | 不变 |
| FieldMapper | widget_id翻译 | widget_id翻译 | 不变 |
| CPM Engine | 前推/后推/约束/级联 | 前推/后推/约束/级联 | 不变 |
| WBS Calculator | 编码计算+拖拽重算 | 编码计算+拖拽重算 | 不变 |
| Resource Scheduler | 冲突检测+决策执行 | 冲突检测（**决策在原生表单完成**） | 精简 |
| Baseline Manager | 版本快照+偏差计算 | 版本快照+偏差计算 | 不变 |
| **🆕 Webhook Receiver** | 无 | **接收智能助手回调，触发重算** | 新增 |
| **~~通用CRUD~~** | 项目/WBS/依赖的增删改查 | **~~删除~~** | 删除 |

| 配置项 | 类型 | 说明 | 工作量 |

### 4.2 关键发现

> **人天总量基本持平，但结构发生了质变。**

| 维度 | V1.0 | V2.0 | 变化 |

| **风险分布** | 集中在二开（CPM+甘特图） | 分散到二开+配置 | 风险更均衡 |
| **维护成本** | 高（全二开） | 中（原生部分免维护） | ↓ 长期维护成本降低 |

### 5.2 原型交互流程变更

**V1.0 流程**（单向）：
```
用户在二开页面操作 → API写入简道云 → 完成
```

**V2.0 流程**（双向闭环）：
```
用户在汇聚页(模式A)发现异常
  → 点击📄链接 → 跳转简道云详情页(模式B)
    → 在详情页编辑字段（原生CRUD）
      → 智能助手检测变更 → Webhook回调Bridge
        → Bridge重算CPM → 写回简道云
          → 用户回到汇聚页刷新 → 看到最新数据 ✅
```

## 八、结论

### V2.0 评估结论

| 维度 | 结论 |
|------|------|
| **可行性** | ✅ Go（与V1.0一致） |
| **推荐报价** | 52-60万（较V1.0下调3-5万） |
| **预估工期** | 120人天中值，2人并行≈60-75天 |
| **核心变化** | 二开占比从53.6%降至42%，更多利用简道云原生能力 |
| **最大收益** | 长期维护成本降低（原生部分免维护）+ 用户体验更统一（简道云原生交互） |
| **新增风险** | 智能助手闭环可靠性（P1）、自定义详情页能力边界（P2） |
